# PR-162: BoundedRecoveryPolicy framework

Branch: pr-162-bounded-recovery-policy
- Type: refactor
- Complexity: medium
- Depends on: none
- Priority: 1
- Coder: any

## Problem

The daemon contains at least two ad-hoc implementations of the same conceptual pattern: count consecutive failure cycles, escalate to a different state when threshold is reached, reset counter on success.

Site 1: dirty-tree auto-recovery in `src/daemon/preflight.py:15-58`. After `_DIRTY_CYCLES_BEFORE_AUTO_RESET = 3` consecutive cycles of a dirty working tree, hard-reset to `origin/{branch}` instead of staying ERROR. Counter `self._consecutive_dirty_cycles` reset on success.

Site 2: FIX iteration cap in `src/daemon/handlers/fix.py`. After `fix_iteration_cap` consecutive FIX cycles on the same PR, escalate via GitHub label and comment, move to IDLE. Counter `current_pr.fix_iteration_count` lives on PRInfo state.

Both sites encode the same logic shape (count → threshold → action → reset). They were written independently, do not share code, and any future failure mode that would benefit from this pattern (no-push FIX deadlock from PR #217 / memory #30, repeated coder-succeeded-no-PR exits, etc.) requires writing the bookkeeping again from scratch.

This refactor extracts the shared shape into a single declarative framework, applies it to the two existing sites without behavior change, and lets future failure modes be added with one declaration instead of a new copy of the bookkeeping.

## Scope

Create `src/daemon/recovery_policy.py` containing the `BoundedRecoveryPolicy` framework. Re-implement the existing dirty-tree and fix-iteration-cap sites on top of the framework. Add unit tests for the framework itself and update existing tests for the two sites.

This PR does NOT add new failure-mode handling. It does NOT change observable daemon behavior — the same thresholds, the same actions, the same counter resets. It is a pure refactor that produces a reusable foundation. New failure modes (FIX no-push deadlock, GitHub rate limit) get separate PRs that USE this framework.

## Files to create

`src/daemon/recovery_policy.py`: the framework module. Sketch (illustrative):

```python
from __future__ import annotations

from dataclasses import dataclass
from typing import Awaitable, Callable, Generic, TypeVar

T = TypeVar("T")


@dataclass(frozen=True)
class BoundedRecoveryPolicy(Generic[T]):
    name: str
    max_attempts: int
    counter_getter: Callable[[T], int]
    counter_setter: Callable[[T, int], None]
    on_threshold: Callable[[T], Awaitable[None]]

    def increment(self, ctx: T) -> int:
        new = self.counter_getter(ctx) + 1
        self.counter_setter(ctx, new)
        return new

    def reset(self, ctx: T) -> None:
        self.counter_setter(ctx, 0)

    async def maybe_escalate(self, ctx: T) -> bool:
        if self.counter_getter(ctx) >= self.max_attempts:
            await self.on_threshold(ctx)
            return True
        return False
```

The generic type `T` is the context object (RepoState, PRInfo, or anything with mutable counter state). `counter_getter` and `counter_setter` are pure accessors so the framework is agnostic to where the counter lives. `on_threshold` runs when the count reaches `max_attempts` and is responsible for whatever the recovery action is (auto-reset, escalate label, transition to HUNG, etc.). After threshold action, the caller is expected to reset the counter explicitly.

`tests/test_recovery_policy.py`: unit tests for the framework. Cover:
- increment increases counter monotonically
- reset zeroes counter
- maybe_escalate returns False below threshold, True at and above threshold
- on_threshold is awaited exactly once when threshold first reached
- counter_getter and counter_setter are called, not direct attribute access

## Files to touch

`src/daemon/preflight.py`: refactor `preflight()` and `_auto_reset_dirty_tree()` to use a `BoundedRecoveryPolicy` instance. The instance is constructed in `__init__` of the daemon runner (or imported as a module-level constant if it does not need runtime config). The counter `self._consecutive_dirty_cycles` continues to exist; the framework just abstracts the increment/reset/threshold logic.

The `_DIRTY_CYCLES_BEFORE_AUTO_RESET = 3` constant becomes the policy's `max_attempts`. The `_auto_reset_dirty_tree()` becomes the policy's `on_threshold` callback.

Final shape (illustrative):

```python
# at module level or in __init__:
self._dirty_tree_policy = BoundedRecoveryPolicy(
    name="dirty_tree_auto_reset",
    max_attempts=3,
    counter_getter=lambda _: self._consecutive_dirty_cycles,
    counter_setter=lambda _, n: setattr(self, "_consecutive_dirty_cycles", n),
    on_threshold=lambda _: self._auto_reset_dirty_tree_async(),
)

# in preflight():
if dirty:
    count = self._dirty_tree_policy.increment(self)
    if await self._dirty_tree_policy.maybe_escalate(self):
        # threshold action ran; check if recovery succeeded
        return True  # or False depending on _auto_reset_dirty_tree result
    # below threshold: stay ERROR as today
    ...
else:
    self._dirty_tree_policy.reset(self)
```

Note: the existing `_auto_reset_dirty_tree()` returns bool synchronously. If it stays sync, wrap it in an async helper or change the on_threshold signature to accept sync OR async callbacks (the framework can adapt). Pick the cleaner option; document the choice in the PR.

`src/daemon/handlers/fix.py`: refactor the fix_iteration_cap escalation block (around the lines that currently check `current_pr.fix_iteration_count >= fix_iteration_cap` and post the comment + label + IDLE transition). Replace with a `BoundedRecoveryPolicy` instance whose:
- counter_getter reads `current_pr.fix_iteration_count`
- counter_setter writes `current_pr.fix_iteration_count`
- max_attempts comes from `self.app_config.daemon.fix_iteration_cap`
- on_threshold runs the existing escalation action (post comment, add label, set state IDLE)

The `current_pr.fix_iteration_count` increment (which currently happens in handle_fix's main flow) gets routed through `policy.increment()` instead. The reset on successful fix gets routed through `policy.reset()`.

`tests/test_preflight.py` and `tests/test_fix_handler.py` (or wherever these are tested today): adjust to verify the new code path. Same observable behavior — same thresholds, same actions — but the call goes through the policy. Existing assertions about resulting state (ERROR after dirty, IDLE after auto-reset, escalated label after fix cap) must still pass without modification.

Add ONE assertion per refactored site that the policy is invoked: mock `BoundedRecoveryPolicy.increment` / `maybe_escalate` and verify call counts. This guards against accidentally bypassing the framework in future edits.
