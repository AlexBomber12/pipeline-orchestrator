## Problem

PR-154a delivered the mock coder shim. This PR adds the first behavioral test that exercises the full PLANNED PR lifecycle through the shim: upload a task zip, observe daemon transition through CODING and WATCH states, observe merge to the testbed main branch, observe daemon return to IDLE. This is the canonical happy-path test and the foundation for diagnosing regressions.

The test uses the shim's "success" scenario. Because shim does real git operations (commit, push, gh pr create), the daemon's GitHub interaction code is exercised end-to-end. Because shim does NOT call any LLM API, the test is deterministic and zero-cost.

This PR adds two new conftest fixtures needed by the test (upload_zip and make_task_zip) plus the test itself. It also adds the reset_testbed fixture in its minimal form (without the "wait for daemon IDLE" robustness — that gets added in a later PR if testing-week edge cases reappear).

## Scope

Add make_task_zip and upload_zip fixtures to tests/e2e/conftest.py. Add a minimal reset_testbed fixture (closes open PRs only, no daemon-state polling). Add tests/e2e/test_upload_and_merge.py with one test.

`tests/e2e/test_upload_and_merge.py`: new file with one test `test_full_happy_path_via_shim_succeeds`.

The test takes fixtures: dashboard_url, testbed_slug, wait_for_state, get_state, upload_zip, make_task_zip, reset_testbed.

Procedure (numbered for the implementer to follow exactly):
1. Wait for daemon IDLE: `wait_for_state(["IDLE"], timeout_sec=30)`. If TimeoutError, the test stack started in an unexpected state; fail with a clear message.
2. Set shim scenario via context manager: `with coder_shim("success"):` (import from tests/e2e/lib/coder_shim.py).
3. Inside the context: build a zip with `zip_path = make_task_zip(800, "e2e-test-upload-merge", coder="any", priority=2)`.
4. Upload: `status = upload_zip(zip_path); assert status in (200, 201)`.
5. Wait CODING: `wait_for_state(["CODING"], timeout_sec=30)`. The daemon's poll cycle is 2 seconds (config.test.yml), so it should pick up the task within 5-10 seconds.
6. Wait WATCH: `wait_for_state(["WATCH"], timeout_sec=90)`. The shim does git push and gh pr create, then daemon transitions to WATCH.
7. Wait MERGE or IDLE: `wait_for_state(["MERGE", "IDLE"], timeout_sec=180)`. With auto_merge=true in config.test.yml and no required CI on testbed, daemon should merge the PR within seconds of WATCH (Codex review will COMMENT or be absent; daemon's merge logic triggers based on CI status which is absent on testbed). If the test ends in IDLE, the merge succeeded and daemon returned to idle. If MERGE is observed transiently, that is fine.
8. Final assertion: `state = get_state(); assert state["state"] == "IDLE"; assert state["current_pr"] is None or state["current_pr"]["state"] == "MERGED"`.

The test does NOT take Playwright `page` fixture and does NOT take screenshots. Screenshots are deferred to PR-154d if behavior requires them. This first test is API-only for simplicity.

The test relies on the testbed repo NOT having branch protection on main and NOT having required CI checks. If those are enabled on testbed, daemon will not auto-merge and the test will time out at step 7. The implementer should verify testbed configuration before debugging — the operator can confirm or fix testbed settings via gh CLI.
