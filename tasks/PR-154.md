# PR-154: Core e2e tests with mock coder shim

Branch: pr-154-e2e-core-tests
- Type: feature
- Complexity: high
- Depends on: PR-153
- Priority: 1
- Coder: any

## Problem

PR-153 set up the integration test infrastructure (docker-compose.test.yml, conftest fixtures, smoke test). This PR populates that infrastructure with the actual behavioral tests that protect the four production behaviors validated empirically during testing-week Days 1 to 5: end-to-end task upload and lifecycle, daemon stop and resume, Redis-down graceful degrade and recovery, and SIGKILL queue integrity preservation.

Two design constraints drive the approach. First, the tests must not burn Claude or Codex quota on every run, so the coder CLIs need to be mocked. Second, the mock must not require modifying src/coders/ (production code) because that increases blast radius and conflicts with PR-156 which is the next planned change to coder selection. The solution is a shim wrapper script that intercepts CLI invocations at the binary level via PATH override, while leaving src/coders/ unchanged.

The shim performs real git operations (commit a placeholder file, push branch, open PR via gh) so the daemon's WATCH and MERGE state machines exercise their actual GitHub interaction code paths, but it does not call any LLM API. Token cost per test run is zero except for negligible GitHub API calls.

## Scope

Add a shim mechanism (a bash wrapper script and a Python context manager) that replaces the claude and codex binaries inside the test stack with deterministic scripts whose behavior is controlled by a SHIM_SCENARIO environment variable or a scenario file. Add four behavioral test files using the shim. Update docker-compose.test.yml minimally to mount the shim scripts. Do not touch src/coders/.

Shim scenarios supported:
- success: shim creates a branch, commits a placeholder edit, pushes, opens a PR with @codex review comment, exits 0. Daemon should observe the PR and progress through WATCH and MERGE.
- no_pr: shim commits and pushes a branch but does not open a PR. Exits 0. Daemon should detect "branch exists, no PR" and increment the retry counter (PR-190a behavior).
- exit_nonzero: shim immediately exits 1 with an error message on stderr. Daemon should treat as transient failure and retry within bounded counter.
- hang: shim sleeps for longer than planned_pr_timeout_sec (60 in test config). Daemon should kill the subprocess and transition to HUNG.
- malformed_pr: shim opens a PR but on a wrong branch name (with prefix mismatch). Daemon should detect and HUNG.
- slow: same as success but with a 30-second delay before commit, used by stop/resume test to give the test time to send Stop while CODING.

The four behavioral tests:
- test_upload_and_merge: full happy path with success scenario. Validates DATA-02 plus the standard PLANNED PR lifecycle.
- test_stop_and_resume: starts a task with slow scenario, clicks Stop mid-CODING, verifies daemon goes to PAUSED or IDLE, clicks Resume via /repos/{slug}/resume, verifies daemon picks the next task.
- test_redis_down_and_recover: starts a task, kills the redis-test container mid-flight, verifies dashboard shows graceful degrade banner, restarts redis-test, verifies daemon recovers to a known state without losing the task.
- test_sigkill_recovery: starts a task with success scenario in CODING state, sends SIGKILL to the daemon-test container, restarts daemon-test, verifies the daemon recovery path correctly identifies the orphan branch and either resumes WATCH on the existing PR or marks the task in a recoverable state. This is the DATA-01 protection.

## Files to create

`tests/e2e/lib/coder_shim.sh`: bash script. Reads scenario from PO_SHIM_SCENARIO_FILE (a path inside the test container, default `/data/shim-scenario`) if it exists, falling back to SHIM_SCENARIO env var, falling back to "success" default. Accepts arbitrary arguments forwarded by the daemon (the real claude and codex CLIs accept variable args) and ignores prompt-related ones. Behavior per scenario:

- success: derive a branch name from the current task by parsing the active DOING entry from `/data/repos/{slug}/tasks/QUEUE.md` (look for the "## PR-{id}:" section followed by "- Status: DOING" and "- Branch: {name}"). cd to `/data/repos/{slug}`. Run `git checkout -b $branch` (or `git checkout $branch` if the local branch already exists). Edit a placeholder file: append a single line "shim marker for {pr_id} at {timestamp}" to a file at `tests/e2e-shim-marker.txt` (creating it if missing) inside the repo. git add the marker file, git commit with message "PR-{id}: shim implementation". git push -u origin $branch. Run `gh pr create --base main --head $branch --title "PR-{id}: shim" --body "Shim PR for testing"`. Run `gh pr comment {pr_url} --body "@codex review"`. Exit 0.
- no_pr: same as success up to and including git push, but skip gh pr create and the comment. Exit 0.
- exit_nonzero: print "shim simulating coder failure" to stderr, exit 1.
- hang: sleep 120 (longer than test config's planned_pr_timeout_sec=60), exit 0 if not killed.
- malformed_pr: same as success but force the branch name to "wrong-branch-prefix-{pr_id}" so daemon's branch validity check fails.
- slow: same as success but with sleep 30 inserted between checkout and commit, giving the test time to send Stop.

The script must be idempotent for git checkout: if the branch already exists locally, switch to it; if it exists on origin, fetch and reset. The script should set `git config user.email "shim@test.invalid"` and `git config user.name "Shim Coder"` before any commit, since the test container may not have global git config.

`tests/e2e/lib/coder_shim.py`: Python module exposing a context manager `coder_shim(scenario)`. On enter, writes the scenario name to `tests/e2e/data/shim-scenario` on the host (which is bind-mounted inside the test container at /data/shim-scenario). On exit, writes "success" back. The shim script reads this file on every invocation. The path mapping must align with TEST_DATA_DIR from PR-153 conftest.

The module also exposes a helper function `verify_shim_installed(compose_file_path)` callable from the test session setup. It runs `docker compose -f {compose_file_path} exec -T daemon-test which claude` and asserts the output is `/usr/local/bin/claude`. If the assertion fails (real binary is shadowed wrongly or shim not mounted), pytest.fail with a clear message pointing at docker-compose.test.yml mounts.

`tests/e2e/test_upload_and_merge.py`: one test `test_full_happy_path_via_shim_succeeds`. Uses fixtures: page, testbed_url, wait_for_state, get_state, upload_zip, make_task_zip, take_screenshot, reset_testbed_clean. Procedure: navigate to testbed_url, wait_for_state IDLE, set shim scenario to success via context manager, build a zip with one trivial task (pr_id e.g. 800, coder=any), upload_zip, wait_for_state CODING within 30s, wait_for_state WATCH within 90s (shim should have opened PR), wait_for_state MERGE or back to IDLE within 180s (auto_merge=true so daemon merges after green CI; testbed CI should pass on a placeholder edit since there are no tests in testbed). Take screenshots at each transition for evidence.

`tests/e2e/test_stop_and_resume.py`: one test `test_stop_during_coding_then_resume`. Uses same fixtures plus dashboard_url. Procedure: setup similar to above, set shim scenario to slow, build zip with pr_id 801, upload, wait_for_state CODING, immediately POST to f"{dashboard_url}/repos/{slug}/stop" with HTMX-compatible headers, wait_for_state IDLE or PAUSED within 30s, verify get_state shows current_task is None or current_pr is None, POST to f"{dashboard_url}/repos/{slug}/resume", set shim back to success, build another zip with pr_id 802, upload, wait_for_state CODING again, wait_for_state IDLE again within 180s (success scenario completes naturally).

`tests/e2e/test_redis_down_and_recover.py`: one test `test_redis_kill_and_restart_recovers`. Uses fixtures. Procedure: wait_for_state IDLE, run `docker compose -f docker-compose.test.yml stop redis-test` from host subprocess, poll `/api/states` for up to 30s expecting either HTTP 200 with PREFLIGHT state or HTTP 200 with degraded indicator (the dashboard layer has graceful read paths per OBS-11). Verify the response body indicates redis unavailable (presence of a banner field, or specific state). Run `docker compose -f docker-compose.test.yml start redis-test`, poll for up to 60s expecting state IDLE and active true. Asserts no daemon-test container restarts occurred (`docker inspect --format '{{.RestartCount}}' pipeline-orchestrator-daemon-test` returns 0).

`tests/e2e/test_sigkill_recovery.py`: one test `test_sigkill_during_coding_recovers_correctly`. Uses fixtures. Procedure: set shim to success, build zip with pr_id 803, upload, wait_for_state CODING, run `docker kill --signal=KILL pipeline-orchestrator-daemon-test`, run `docker compose -f docker-compose.test.yml up -d daemon-test` to bring it back, poll get_state for up to 90s. The expected recovered state depends on how far CODING got: if the shim already pushed a branch, daemon should recover into WATCH on the existing PR; if shim was killed before push, daemon should re-trigger CODING. Either is acceptable; the test asserts state is in IDLE, CODING, or WATCH after recovery, never ERROR or HUNG, and current_task matches the uploaded pr_id 803.

## Files to touch

`docker-compose.test.yml` (created in PR-153): add a volume mount for the shim wrapper. Mount `./tests/e2e/lib/coder_shim.sh:/usr/local/bin/claude:ro` and `./tests/e2e/lib/coder_shim.sh:/usr/local/bin/codex:ro` on both daemon-test and web-test services. The /usr/local/bin path takes precedence over /usr/bin/claude (real binary) because of standard PATH ordering in the python:3.12-slim base image. Verify by running `docker compose -f docker-compose.test.yml exec daemon-test which claude` after stack restart; expected output `/usr/local/bin/claude`.

If the precedence assumption does not hold, instead set the daemon-test PATH env var explicitly: `PATH=/usr/local/bin:${PATH}` in the environment block. This is the safer alternative.

Also pass through environment variable PO_SHIM_SCENARIO_FILE=/data/shim-scenario to both services so the shim script knows where to read the scenario.

`tests/e2e/conftest.py` (created in PR-153): add a session-scoped autouse fixture `_verify_shim` that calls verify_shim_installed at session start. This catches misconfiguration early (test stack up but shim not mounted) with a clear error rather than mysterious test failures later.

## Success criteria

1. `pytest tests/e2e/ -v` with the test stack running collects exactly 5 tests (1 from PR-153 smoke, 4 new) and all 5 pass.

2. Total wall-clock for `pytest tests/e2e/` is under 10 minutes on a typical homelab machine. Individual tests should be 30 to 180 seconds each.

3. After the full suite completes, `gh pr list -R AlexBomber12/pipeline-orchestrator-testbed --state open` returns no open PRs (reset_testbed_clean works correctly).

4. After the suite completes, the testbed repo's main branch may contain merged shim commits if test_upload_and_merge ran to completion (auto_merge=true in test config). This is acceptable; testbed is dedicated to test runs and may have a periodic main-branch reset workflow (out of scope for this PR; documented as known limitation in tests/e2e/README.md).

5. Running the suite twice in succession works (no state leakage between runs).

6. With shim scenario forced to exit_nonzero, test_upload_and_merge fails with a TimeoutError at wait_for_state(CODING, 30s) because the daemon retries and eventually HUNGs; the test correctly identifies this as failure rather than passing silently. This is verified manually before merge by temporarily editing the test to use exit_nonzero and confirming it fails.

7. `scripts/ci.sh` continues to exit 0 with no integration tests collected (norecursedirs config from PR-153 still in effect).

8. ruff check src tests passes.

9. `git status` after suite completion shows no tracked-file changes; only gitignored tests/e2e/data/ and tests/e2e/evidence/ may have content.

10. Test screenshots in tests/e2e/evidence/ are captured for at least the upload_and_merge happy path with descriptive filenames including state name and timestamp.

11. The shim script does not import or depend on any LLM API; running the shim with no network access (offline) for everything except gh CLI calls succeeds. This is verifiable by `docker compose -f docker-compose.test.yml exec daemon-test sh -c 'unset GITHUB_TOKEN; SHIM_SCENARIO=success /usr/local/bin/claude'` which should fail at the gh push step but not at any LLM call.
