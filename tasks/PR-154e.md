# PR-154e: e2e test 4 — SIGKILL daemon recovery

- Type: feature

- Priority: 1
- Coder: any

## Problem

PR-154b validated happy path. PR-154c covered Stop/Resume. PR-154d covered Redis failure. This PR adds the fourth and final behavioral test: SIGKILL the daemon container during CODING, then verify the daemon recovers to a sensible state on restart. This protects DATA-01 (queue integrity under hard kill) which was empirically validated during testing-week Day 1.

The test is the strictest of the four because hard-kill bypasses any graceful shutdown paths. The daemon must reconstruct state from durable sources (GitHub PRs, file system QUEUE.md and tasks/) on restart, NOT from in-memory state which is lost. Specifically the daemon must not re-trigger CODING for a task whose PR is already open (that would create a duplicate branch); it must transition into WATCH on the existing PR, or into IDLE if the task is already complete.

## Scope

Add tests/e2e/test_sigkill_recovery.py with one test. No fixtures, no shim, no compose changes. The shim's "slow" scenario (added in PR-154c) is reused to hold daemon in CODING long enough for the SIGKILL.

## Files to create

`tests/e2e/test_sigkill_recovery.py`: one test `test_sigkill_during_coding_recovers_correctly`.

Fixtures used: dashboard_url, testbed_slug, wait_for_state, get_state, upload_zip, make_task_zip, reset_testbed.

Procedure:
1. Wait for daemon IDLE: `wait_for_state(["IDLE"], timeout_sec=30)`.
2. Open `coder_shim("slow")` context. Inside:
3. Build zip with pr_id=804, upload, assert HTTP success.
4. Wait CODING: `wait_for_state(["CODING"], timeout_sec=30)`.
5. SIGKILL daemon container: `subprocess.run(["docker", "kill", "--signal=KILL", "pipeline-orchestrator-daemon-test"], check=True, timeout=10)`.
6. Verify the daemon container is stopped: `subprocess.run(["docker", "inspect", "--format", "{{.State.Running}}", "pipeline-orchestrator-daemon-test"], ...)` returns "false". Also capture the ExitCode field; it should be 137 (128 + 9 = SIGKILL exit code).
7. Restart daemon: `subprocess.run(["docker", "compose", "-f", "docker-compose.test.yml", "up", "-d", "daemon-test"], check=True, timeout=60)`.
8. Wait for daemon to be running again: poll `docker inspect ... {{.State.Running}}` for up to 30 seconds, expect "true".
9. Wait for daemon to reach a recovered state: `recovered = wait_for_state(["IDLE", "CODING", "WATCH"], timeout_sec=90)`. Either of these three is acceptable depending on how far CODING got before the kill:
   - IDLE: shim was killed before pushing anything; daemon sees no open PR and no DOING task; returns to IDLE.
   - CODING: daemon may decide to re-trigger CODING if it cannot recover prior state; this is acceptable as long as it does not loop.
   - WATCH: shim already pushed the branch and opened a PR before being killed; daemon sees the open PR and resumes WATCH on it.
10. Assert the recovered state is NOT in {"ERROR", "HUNG"}: this would indicate daemon could not recover and is stuck.
11. Verify current_task tracking is consistent: if recovered state is CODING or WATCH, then `current_task["pr_id"] == 804`. If recovered state is IDLE, current_task is None.

## Success criteria

1. With the test stack up and shim mounted, `pytest tests/e2e/test_sigkill_recovery.py -v` collects one test and passes within 4 minutes.

2. The test passes deterministically across at least three consecutive runs. The recovered state may differ between runs (IDLE / CODING / WATCH) depending on exact timing of the SIGKILL, but the assertion in step 10 (not ERROR or HUNG) holds reproducibly.

3. The daemon container's ExitCode after SIGKILL is 137 (verified in step 6).

4. After the test, the daemon container is running and registered with the testbed repo. `get_state()` returns a valid dict.

5. After the test, `gh pr list -R AlexBomber12/pipeline-orchestrator-testbed --state open` may show PR-804 if the shim opened it before the kill; reset_testbed teardown handles this.

6. Full e2e suite `pytest tests/e2e/ -v` collects 5 tests (smoke, upload_and_merge, stop_and_resume, redis_down, sigkill) plus the shim verification fixture, and all pass within 15 minutes total.

7. With shim scenario forced to "success" instead of "slow" in step 2 (manual modification), the test would race: SIGKILL might fire after the shim already finished. The test would still likely pass because step 9 accepts IDLE as a recovered state, but it would not exercise the harder "kill during in-flight" path. The "slow" choice is intentional.

8. `python -m pytest -q` continues to exclude tests/e2e/. scripts/ci.sh exits 0.

9. ruff check src tests passes.

10. `git status` after test runs shows no tracked-file modifications.

11. Every `subprocess.run` call uses an explicit timeout argument. No bare invocations.

12. The test does NOT assume any particular recovery strategy; it accepts three valid recovered states (IDLE, CODING, WATCH). This is critical because the daemon's recovery code is allowed to evolve in future PRs (e.g. PR-156 onwards may change WATCH behavior); this test should not need updating unless the recovery contract itself changes.
