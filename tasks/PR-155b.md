## Problem

PR-155 introduces GitHub Actions as the authoritative integration test gate. After PR-155, every PR is automatically validated by the integration job, and the coder no longer needs to run e2e tests locally. However, two cases still require local execution:

First, debugging a failing GHA integration run. The GHA logs and uploaded e2e-evidence artifact help, but reproducing the failure interactively on a developer machine is faster than blind iteration via "push, wait 15 minutes, read logs, push again". A wrapper script that brings up the test stack, runs a single test by name, keeps the stack up for inspection, and tears down on demand cuts that loop substantially.

Second, developing new tests. Writing a new test_xxx.py file requires running it in isolation many times. Doing this through GHA is wasteful (unrelated jobs queue, runner minutes consumed) and slow (push cycle).

This PR adds a host-side wrapper script and the documentation explaining when to use it and how to debug common failures. It does not change the merge contract from PR-155: GHA still gates merges. This is purely developer ergonomics for the cases above.

## Files to create

`scripts/test-e2e.sh`: bash script with `set -euo pipefail`. The script's first three lines after the shebang are a comment block describing usage in plain text (no special markdown).

Behavior:
- cd to the repo root (resolved via `cd "$(dirname "$0")/.."`)
- Parse flags via while/case loop over $@. Recognized flags:
  - `--keep-up`: skip teardown after tests (leaves stack running for debugging)
  - `--no-up`: skip startup (assume stack already up)
  - `--filter PATTERN`: forward as `-k PATTERN` to pytest
  - `--scenario NAME`: pre-set the shim scenario file to NAME before running tests (default behavior is "success" via context managers in tests, but this allows manual override for debugging)
  - `--help` or `-h`: print usage and exit 0
- If not --no-up: run `docker compose -f docker-compose.test.yml up -d --wait`. Capture failure if compose times out.
- Optionally, if --scenario provided: write that string to tests/e2e/data/shim-scenario.
- Run `python -m pytest tests/e2e/ -v ${FILTER_FLAG:-}` and capture exit code.
- If not --keep-up: run `docker compose -f docker-compose.test.yml down -v` regardless of pytest exit code (always teardown if user did not explicitly request keep).
- Echo a one-line summary indicating PASS or FAIL and total wall-clock duration (use `SECONDS` builtin).
- Exit with the captured pytest exit code.

The script makes no GitHub Actions assumptions and works on AI-Server, on DESKTOP-5NT9DG3 WSL, and on any machine with Docker, Python, gh, and the test stack files. It does NOT generate App tokens; locally the developer's own gh CLI auth (with personal credentials) is used. This means local runs use the developer's gh permissions, while GHA runs use the App. Both are valid against testbed.

`docs/local-e2e.md`: markdown document, target length 100 to 150 lines.

Sections in order:

1. Overview. State that local e2e is for debugging GHA failures and for developing new tests, NOT for routine merge gating (GHA owns that). Link to docs/ci-setup.md for the GHA setup itself.

2. Prerequisites. Docker and Docker Compose v2 (verified `docker compose version` returns v2.x), Python 3.12, gh CLI authenticated (`gh auth status` exits 0), Playwright browsers installed (`playwright install chromium` after pip install -r requirements-test.txt). Note that on AI-Server and DESKTOP-5NT9DG3 WSL these are already configured per the homelab runbook.

3. Quick start. Single command examples:
   - `scripts/test-e2e.sh` runs the full e2e suite with stack up and teardown
   - `scripts/test-e2e.sh --filter test_upload_and_merge` runs only that test
   - `scripts/test-e2e.sh --keep-up` for inspecting state after tests
   - `scripts/test-e2e.sh --no-up --keep-up --filter test_smoke` for fastest iteration when actively developing

4. Architecture. Diagram in plain ASCII showing the port and namespace separation: prod stack on port 8800 with bind mount ./data/, test stack on port 18800 with bind mount ./tests/e2e/data/, completely independent docker networks. Both can run simultaneously on the same host without conflict.

5. The mock coder shim. Reference tests/e2e/lib/coder_shim.sh from PR-154. List the six scenarios (success, no_pr, exit_nonzero, hang, malformed_pr, slow). Explain that the shim does real git operations (commit, push, gh pr create) but no LLM calls, so test runs cost zero in tokens.

6. Reproducing a GHA failure locally. Steps:
   - Open the failing GHA run in browser
   - Note the failing test name and the assertion or timeout that fired
   - Download the e2e-evidence artifact zip; extract screenshots and stack-logs.txt
   - On local machine, run `scripts/test-e2e.sh --filter <test_name> --keep-up`
   - If reproduces locally, debug interactively (docker compose logs daemon-test, web-test, redis-test from a second terminal; curl http://localhost:18800/api/states; etc.)
   - If does NOT reproduce locally, suspect environmental difference: Python version, Playwright version, network conditions. Document the divergence.

7. Troubleshooting common failures.
   - Port 18800 already in use: another test stack is up. Run `docker ps | grep test`. Kill stale stacks with `docker compose -f docker-compose.test.yml down -v`.
   - Tests fail with auth errors at the gh push step: check `gh auth status` returns logged in to AlexBomber12 with repo scope. If not, run `gh auth login`.
   - Tests fail with "permission denied" on /data/auth/.codex inside daemon-test: the codex-auth-test named volume needs initialization. Run `docker run --rm -v pipeline-orchestrator_codex-auth-test:/target alpine chown -R 1000:1000 /target` once.
   - Tests hang at wait_for_state(IDLE): testbed repo has open PRs from a previous run. Run `gh pr list -R AlexBomber12/pipeline-orchestrator-testbed --state open --json number --jq '.[].number' | xargs -I {} gh pr close {} -R AlexBomber12/pipeline-orchestrator-testbed --delete-branch`.
   - Playwright "browser not found": run `playwright install chromium` in the venv where pytest runs.
   - Test passes locally but fails in GHA: check that the GHA run uses the same Python version (3.12) and same Playwright version. Differences in browser version are common; pin in requirements-test.txt if drift causes flakes.

8. Why this is separate from scripts/ci.sh. The technical reason is that scripts/ci.sh runs inside the daemon container as part of the coder's pre-PR workflow. The daemon container has no access to the host Docker socket (intentional security boundary; mounting it would give LLM-driven coder code the ability to manipulate any container on the host). Without socket access, ci.sh cannot start docker compose for the test stack. Integration tier therefore runs from a host-level entry point: this script for local invocation, and GitHub Actions runners for automated gating. The two share the same docker-compose.test.yml and conftest fixtures but differ in entry point and credential mechanism (local uses developer gh auth, GHA uses App-generated token).

9. Adding new tests. Pattern: copy tests/e2e/test_smoke.py as starting template, request fixtures from conftest.py (do not duplicate setup logic), name the file test_xxx.py so pytest discovers it, run via `scripts/test-e2e.sh --filter test_xxx` until green, push, and let GHA validate in clean environment. If a new shim scenario is needed, add it to tests/e2e/lib/coder_shim.sh first then a test that invokes coder_shim("new_scenario").
