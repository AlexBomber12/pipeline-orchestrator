# PR-146: Pause and Stop event log feedback with cancel-pause support

Branch: pr-146-pause-stop-events
- Type: ux
- Complexity: low
- Depends on: none
- Priority: 2
- Coder: any

## Problem

The three control endpoints in `src/web/app.py` (`/repos/{name}/pause`, `/resume`, `/stop`) flip Redis state and return JSON with zero event log feedback at click time. Operator clicks Pause, sees no acknowledgment until daemon reaches a checkpoint and emits a terse technical message like "User pause persisted during coder exit; finishing current run before honoring pause."

Missing user-facing feedback:
1. Immediate acknowledgment on click (pause, resume, stop).
2. Distinction between "Play cancels a pending pause" vs "Play resumes from paused-IDLE" - both hit the same endpoint with identical handling.
3. Human-readable "Paused" event at the effective transition point (daemon in paused-IDLE, not picking next task).

The existing daemon-side "Paused by user, not picking up new tasks" event IS emitted exactly once per pause-effective transition via the `_user_pause_logged` flag on the runner (lifecycle in `src/daemon/runner.py` lines 221, 817, 887 reset; 859, 878 and `src/daemon/handlers/idle.py:645` emit). Debounce infrastructure is correct. Only the wording is wrong. Do not invent new infrastructure.

## Scope

### Five agreed event texts

- Pause clicked while daemon active: `Pause requested. Finishing current PR cycle (may take several iterations).`
- Play clicked while user_paused=True AND daemon is in an active state (CODING/WATCH/FIX/MERGE): `Pause canceled. Continue current run.`
- Play clicked while user_paused=True AND daemon is in IDLE or PAUSED state: `Resumed. Taking next task from queue.`
- Daemon reaches paused-IDLE effective transition (first emission debounced by `_user_pause_logged`): `Paused. Press Play to resume.`
- Stop clicked: `Stop requested. Aborting run; working tree may be left dirty.`

### Endpoint events (new)

In `src/web/app.py`:
- `pause_repo`: after `_update_repo_pause_state` succeeds, emit Pause text.
- `resume_repo`: inside or after the transaction, read current state.state value. Emit "Pause canceled" text if state is in CODING/WATCH/FIX/MERGE; emit "Resumed" text otherwise.
- `stop_repo`: after `_update_repo_pause_state` succeeds, emit Stop text.

Emissions from endpoint code must use the same event-log mechanism the daemon uses (write to Redis via the same helper, so it shows in `repo.history` and SSE stream). If emitting from web process requires a new helper to wrap the event append, add it as part of this PR in `src/web/app.py` or a thin wrapper module; do not touch the daemon-side event writer.

### Existing emission text change

In all three daemon emission sites (`src/daemon/runner.py:861`, `src/daemon/runner.py:880`, `src/daemon/handlers/idle.py:647`), change the text from `"Paused by user, not picking up new tasks"` to `"Paused. Press Play to resume."` Do not change the debounce logic or the flag.

The other pause-related messages ("Pause requested while preparing ... deferring CODING", "User pause persisted during coder exit; finishing current run before honoring pause", "User stop requested ...", "CODING aborted: user stop requested", "FIX aborted: user stop requested") remain as-is. They fire in specific internal transitions that are useful for debugging and do not conflict with the user-facing five.
