# PR-057: Decompose runner.py into focused modules

Branch: pr-057-runner-decomposition

- Type: architecture
- Complexity: high

## Problem

`src/daemon/runner.py` is 2253 lines with 37 functions: state machine handlers, git helpers, recovery logic, queue parsing, rate limit detection, queue-sync remediation, codex review posting.

Two concrete impacts:

1. Claude Code in --print mode loads this file into context on every CODING/FIX invocation. Larger context = more memory = more exposure to the upstream memory leak. Smaller modules = lower OOM risk.

2. Codex reviews 2253-line files less reliably than focused 200-400 line modules. Review quality degrades with file size.

All sprint 7 work ahead of this PR (PR-047 through PR-056) modifies handle_error, handle_idle, handle_fix, and other handlers. Placing decomposition last minimizes merge churn - other PRs edit the monolith, this one splits the final result.

## Scope

Pure refactor. Split runner.py into focused modules using mixin pattern. No behavioral changes. Public API of PipelineRunner unchanged.

Target: no module exceeds 400 lines.

## Approach

Mixin classes. Each handler module defines a mixin class. PipelineRunner inherits from all of them.

Pros over module-level function pattern:
- Methods stay as methods, self access unchanged
- No signature changes across the codebase
- Tests that patch PipelineRunner methods continue to work
- Lower-risk refactor

Cons:
- Multiple inheritance hides method origin in some IDEs
- Mixin order matters for method resolution (document it)

Mixin order: less-dependent mixins first (recovery, preflight, rate_limit, repo_ops). Handler mixins last since they may call methods from earlier mixins via self.

## Changes

### For each new module

1. Create file with imports (from typing, datetime, asyncio, etc.)
2. Define mixin class with extracted methods
3. Preserve exact method body and decorators
4. Add module docstring explaining which mixin methods belong here

### src/daemon/runner.py

- Remove extracted method bodies
- Add mixin imports and inheritance
- Keep PipelineRunner.__init__ fully intact (it initializes all the instance variables the mixins reference)
- Keep run_cycle, log_event, publish_state, _save_cli_log, _publish_while_waiting as methods on PipelineRunner

### tests

Existing tests import PipelineRunner and call its methods. They should pass unchanged because PipelineRunner's external interface is preserved.

Update only:
- Module-level helper imports: `from src.daemon.runner import _git` -> `from src.daemon.git_ops import _git`
- Mock paths for module-level functions: `patch("src.daemon.runner._git")` -> `patch("src.daemon.git_ops._git")`
- Mock paths for _TRANSIENT_STATES and similar constants if they move

Run scripts/ci.sh after each module extraction to catch regressions early. If ci fails, commit the partial refactor before proceeding so there's a clean bisection point.

## Execution strategy

Recommended order within this PR (multiple commits):
1. Extract git_ops.py (module-level functions, lowest risk)
2. Extract rate_limit.py
3. Extract preflight.py
4. Extract repo_ops.py (ensure_repo_cloned etc)
5. Extract recovery.py
6. Create handlers/ directory, extract one handler at a time
7. Finally update PipelineRunner class definition with mixin inheritance

After each step, run scripts/ci.sh. Commit atomically per extraction.

## Success criteria
- scripts/ci.sh exits 0
- All existing tests pass without behavioral modification
- runner.py is under 400 lines
- No file in src/daemon/ exceeds 400 lines (excluding scaffolder.py which is unchanged)
- PipelineRunner public API unchanged (daemon/main.py works without modification)
- Mixin pattern used consistently across handler modules
- git_ops.py uses module-level functions (not class methods) since those are already stateless
- Module docstrings explain which mixin methods belong where
