# PR-165: GitHub state polling during long FIX cycle

Branch: pr-165-poll-github-during-fix
- Type: feature
- Complexity: medium
- Depends on: PR-163
- Priority: 2
- Coder: any

## Problem

During a FIX cycle, daemon spawns the coder process (claude / codex CLI) and waits synchronously for it to exit. The coder may take 5-30+ minutes for a single FIX iteration, depending on task complexity and CLI quota. While the coder runs, daemon does NOT poll GitHub for state changes.

Two specific bad outcomes were observed on PR #217 on 2026-04-26 (memory entry #30):

1. **Missed external merge.** User merged PR #217 manually via gh pr merge while claude was running a 30-minute FIX cycle. Daemon was unaware, kept the coder running, then transitioned WATCH→FIX→WATCH on its return. Wasted wall clock; user had to restart the container.

2. **Missed external close / abandon.** If user closes the PR externally (via web UI, gh pr close), daemon does not notice until the FIX cycle ends. Coder continues working on a now-abandoned PR.

Both outcomes are observability gaps. The daemon's "I'm in FIX" state should not be a black hole; the daemon should periodically check whether the work it is doing is still relevant.

The fix: while a coder process is running, periodically poll GitHub state in a side task. If the PR is merged, closed, or otherwise reaches a terminal state, signal the coder process to terminate and transition daemon to the appropriate next state.

## Scope

Add a background polling task to the FIX handler that runs concurrently with the coder subprocess. On each poll, check PR state. If terminal (merged, closed), gracefully terminate the coder and transition daemon. Use the BoundedRecoveryPolicy framework from PR-162 to manage the polling cadence and any failure modes. Inherit GitHub API budget awareness from PR-163.

This PR does NOT change FIX semantics on the productive path. It does NOT introduce new state transitions that didn't already exist. It adds an early-termination capability that converts a previously-blind wait into an observable wait.

## Files to touch

`src/daemon/handlers/fix.py`: at the point where coder subprocess is launched (around line 462 where `_save_cli_log` is called), wrap the subprocess wait in `asyncio.gather` with a polling task:

```python
poll_task = asyncio.create_task(self._poll_github_during_fix(pr_number))
try:
    coder_result = await self._run_coder_with_polling(coder, prompt, repo_path, poll_task)
finally:
    poll_task.cancel()
```

`_poll_github_during_fix(pr_number)`:
- Loops every `app_config.daemon.fix_poll_interval_sec` seconds (default 30).
- Each iteration: fetch PR state via `gh pr view {number} --json state,mergedAt,closedAt,statusCheckRollup`.
- If state is `MERGED` or `CLOSED`: log "PR #{number} reached terminal state {state} during FIX, requesting coder termination." Set a flag readable by the wait coroutine.
- If gh fails (rate limit, network), log warning, continue polling. Do NOT crash the daemon.
- Cancellation cleanup: when poll_task is cancelled (FIX completes normally), exit cleanly.

`_run_coder_with_polling(coder, prompt, repo_path, poll_task)`:
- Runs the coder subprocess as today (existing logic).
- Concurrently checks if poll_task signals terminal state.
- If terminal state detected, sends SIGTERM to coder subprocess, waits up to `app_config.daemon.coder_terminate_grace_sec` (default 5) for graceful exit, then SIGKILL if still running.
- Returns appropriate result so handle_fix knows whether to transition to IDLE (merged), HUNG (closed externally), or normal post-FIX flow (coder completed naturally).

`config.yml`, `config.test.yml`:
- `fix_poll_interval_sec: 30` — how often to check PR state during FIX
- `coder_terminate_grace_sec: 5` — SIGTERM timeout before SIGKILL

`src/daemon/handlers/fix.py` (state transition extension): on detected external merge:
- Log "PR #{number} merged externally during FIX, returning to IDLE."
- Reset `no_push_fix_count` (introduced in PR-164) and `fix_iteration_count`.
- Transition state to IDLE.
- Trigger _mark_queue_done logic if applicable.

On detected external close:
- Log "PR #{number} closed externally during FIX, transitioning to HUNG."
- Transition to HUNG. User can resolve manually.

`tests/test_fix_handler.py`: add tests covering:
- Coder runs to completion normally → poll task cancelled, no transition
- Coder running, GitHub poll detects MERGED → SIGTERM sent, coder exits, daemon transitions IDLE
- Coder running, GitHub poll detects CLOSED → SIGTERM sent, daemon transitions HUNG
- gh poll fails (rate limit), poll task continues without crashing
- Poll task cancellation cleanup is clean (no asyncio warnings)

`tests/e2e/test_fix_external_merge.py` (new): end-to-end test where a PR is created, daemon enters FIX, test externally merges via gh pr merge, verify daemon detects within poll interval and transitions to IDLE.
