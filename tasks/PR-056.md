# PR-056: Review status parser strictness

Branch: pr-056-review-status-parser

- Type: bugfix
- Complexity: low

## Problem

After PR-047 removes the P1/P2 substring filter from CHANGES_REQUESTED detection, the parser is more permissive: any Codex comment after the anchor flips the PR to CHANGES_REQUESTED. This is correct for the common case, but the APPROVED check in `_compute_review_status` still relies on loose matching of the +1 reaction content field.

Residual risk: if Codex introduces a new reaction naming convention, or a summary comment like "No P1/P2 issues found" is posted without the +1 reaction, the classifier could produce wrong results in edge cases.

This PR tightens the positive APPROVED detection with stricter matching on reaction content and author.

## Scope

Harden the +1 detection:
- Match only exact reaction content "+1", not substring
- Require the reacting user's login to match a known Codex bot pattern
- Prefer structured review format markers when available (review state APPROVED/CHANGES_REQUESTED from GitHub review API) as a stronger signal than comment reactions

Does not change the PENDING/CHANGES_REQUESTED logic fixed in PR-047 and PR-048.

## Changes

### src/github_client.py

Add a helper for strict Codex bot identification:

```python
CODEX_BOT_LOGIN_PATTERN = re.compile(
    r"(^|[^a-z])codex($|[^a-z])", re.IGNORECASE
)

def _is_codex_user(user_dict: dict | None) -> bool:
    """Return True if the GitHub user object represents a Codex bot."""
    if not isinstance(user_dict, dict):
        return False
    login = user_dict.get("login", "") or ""
    return bool(CODEX_BOT_LOGIN_PATTERN.search(login))
```

Replace loose `"codex" in login.lower()` checks with `_is_codex_user(reaction.get("user"))` throughout `_compute_review_status` and related helpers.

Tighten reaction content matching:

```python
def _is_plus_one(reaction: dict) -> bool:
    """Return True if the reaction is exactly +1 from a Codex user."""
    if not isinstance(reaction, dict):
        return False
    if reaction.get("content") != "+1":
        return False
    return _is_codex_user(reaction.get("user"))
```

Use this helper in `_find_codex_plus_one_reaction` and body-approved detection.

### Optional: use GitHub review API state

GitHub's `/repos/{repo}/pulls/{pr_number}/reviews` endpoint returns structured review objects with a `state` field: APPROVED, CHANGES_REQUESTED, COMMENTED, DISMISSED. If any Codex-authored review has state APPROVED and its submitted_at is after the latest push, that is a stronger signal than reaction-based detection.

Add a fallback layer:
1. Check for Codex review with state=APPROVED submitted after latest push -> APPROVED
2. Otherwise fall through to existing reaction-based logic

This only adds detection, does not remove existing paths. Low risk.
