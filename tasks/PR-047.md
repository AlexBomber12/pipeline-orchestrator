# PR-047: Fix review status PENDING for any Codex comment after anchor

Branch: pr-047-review-status-pending

- Type: bugfix
- Complexity: low

## Problem

Two classification points in the codebase filter Codex comments by "P1" or "P2" substring in body. When Codex posts feedback without those tags (e.g. "Looks fine, consider X" or stylistic suggestions), the daemon classifies the PR as PENDING instead of CHANGES_REQUESTED. handle_watch routes PENDING into waiting state, eventually escalating to HUNG after review timeout.

Also affects recovery: if recover_state runs while a PR is in this state, runner stays in WATCH and never makes progress.

## Root cause

Two functions enforce the same incorrect filter:

1. `src/github_client.py:285-298` in `_compute_review_status` only returns CHANGES_REQUESTED when a Codex comment after the anchor contains "P1" or "P2". Otherwise falls through to PENDING.

2. `src/daemon/runner.py:2196-2209` in `_has_new_codex_feedback_since_last_push` only counts comments with "P1" or "P2" as new feedback. Even if github_client returned CHANGES_REQUESTED, runner would still skip FIX if comments lack severity tags.

PENDING should mean "Codex has not reviewed yet at all". Any Codex activity after the anchor without an approval signal should trigger FIX.

## Scope

Fix both classification points so that:
- Codex comment after anchor (any content) without +1 -> CHANGES_REQUESTED
- Codex comment after last push (any content) -> new feedback, trigger FIX
- No Codex activity after anchor, no reactions -> PENDING
- Existing APPROVED and EYES paths unchanged

## Changes

### src/github_client.py

In `_compute_review_status` around lines 284-298, replace the P1/P2 substring scan with a generic Codex-comment-after-anchor check. The new logic returns CHANGES_REQUESTED when any Codex user has posted a comment after the anchor timestamp, regardless of body content. APPROVED check stays first so a valid +1 still wins over subsequent feedback.

Order of returns:
1. APPROVED (if body_approved or anchor_approved)
2. CHANGES_REQUESTED (if any Codex comment after anchor_ts)
3. PENDING (no Codex activity after anchor)

### src/daemon/runner.py

In `_has_new_codex_feedback_since_last_push` around lines 2196-2210, remove the P1/P2 filter. Any Codex comment posted after `self._last_push_at` counts as new feedback. Severity tag prioritization happens at the FIX execution layer based on AGENTS.md rules, not in the gating check.

Update the docstring: "Returns True if Codex posted any comment (review or issue) after self._last_push_at. P1/P2 prioritization is handled by Claude in FIX REVIEW mode."

### tests/test_github_client.py

Add tests:
- test_review_status_changes_requested_without_p1_p2_tags: Codex comment "Looks fine, consider X" after anchor, no reactions, expect CHANGES_REQUESTED
- test_review_status_pending_when_no_codex_activity: no comments after anchor, no reactions, expect PENDING
- test_review_status_approved_wins_over_codex_comment: valid +1 reaction plus Codex comment after anchor, expect APPROVED
- test_review_status_eyes_wins_over_codex_comment: eyes reaction plus Codex comment after anchor, expect EYES

Existing P1/P2 tests continue to pass since P1/P2 comments are now a subset of "any Codex comment after anchor".

### tests/test_runner.py

Add tests for `_has_new_codex_feedback_since_last_push`:
- test_has_new_feedback_returns_true_for_any_codex_comment_after_push: comment without P1/P2 posted after _last_push_at, expect True
- test_has_new_feedback_returns_false_for_old_comments: comment posted before _last_push_at, expect False
- test_has_new_feedback_ignores_non_codex_users: comment from regular user after _last_push_at, expect False

Existing P1/P2-only tests continue to pass.
