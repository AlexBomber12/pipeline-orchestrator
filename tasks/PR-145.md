# PR-145: FIX iteration cap with ESCALATE at iteration boundary

Branch: pr-145-fix-iteration-cap
- Type: feature
- Complexity: medium
- Depends on: PR-144
- Priority: 1
- Coder: any

## Problem

The FIX state has no hard upper bound on iterations. PR-138 during the recent sprint reached 10 push iterations as a valid long-tail PR. Without a cap:

1. A genuinely stuck loop (code fails CI, coder pushes fix, CI fails the same way) consumes tokens and wall time indefinitely.
2. Once Pause is honored by the daemon, worst-case Pause latency is unbounded - the user clicks Pause, then waits hours for a looping FIX to finish.
3. Cost per merged PR has no ceiling, breaking the core metric of the product.

Cap of 15 is generous: observed max was 10, so the cap triggers only on real runaway, not on normal long-tail work.

A secondary issue: `PRInfo.push_count` currently has mixed semantics. It is initialized from `len(commits)` (GitHub API) in `src/github_client.py:190` but then incremented daemon-side in `src/daemon/handlers/error.py:271` and `src/daemon/handlers/fix.py:368`. Using it as the iteration counter for cap purposes would be fragile (a PR with N existing commits plus daemon increments would cap earlier than intended). Introduce a dedicated `fix_iteration_count` field to decouple.

### Note on the PR-144 dependency

PR-144 (bounded subprocess wait after TimeoutError/CancelledError) is listed as a hard dependency for safety, not because the cap logic itself cancels anything. Cap check runs at iteration boundaries only. However, Pause-during-FIX and Stop-during-FIX paths in the daemon still rely on `asyncio.Task.cancel()` and `_terminate_current_coder`, which in turn call `proc.wait()` inside the CLI wrapper. Without PR-144's bounded wait, those cancellations can hang on a zombie process. Landing PR-144 first ensures cap-reached escalate does not race into a pre-existing cancel-path bug during operator-driven Pause testing.

## Scope

### New field

Add `fix_iteration_count: int = 0` to `PRInfo` in `src/models.py`. Persisted via Redis as part of RepoState serialization (already handled by pydantic model dump).

### Increment

In `src/daemon/handlers/fix.py` around line 368, alongside the existing `push_count += 1` increment, also increment `fix_iteration_count`. Use `fix_iteration_count`, not `push_count`, as the iteration number passed to downstream logic (the existing `iteration = self.state.current_pr.push_count` at line 370 should be migrated to `fix_iteration_count`). Counter resets to 0 when `current_pr` transitions to a new PR (existing PR-transition logic).

### Cap check at iteration start

Before spawning the next coder in FIX state, check `fix_iteration_count >= fix_iteration_cap`. If true:

1. Do NOT spawn the coder for this iteration.
2. Post comment on the PR: `@AlexBomber12 FIX iteration cap reached ({count}/{cap}). Escalating for manual review.`
3. Add label `escalated` on the PR (create the label via `gh label create escalated --color B60205 --description "Daemon escalated, manual review required"` if absent; ignore failure if label already exists).
4. Transition daemon state to IDLE without touching `user_paused`.
5. Log event: `FIX cap reached ({count}/{cap}) on PR #{pr_number}: escalated, moving to IDLE.`

No mid-work cancellation. Current in-flight coder (if any) runs to natural completion before the check fires on the next iteration boundary.

### Config

- `fix_iteration_cap: int = 15` in the daemon config model
- Env var override `PO_FIX_ITERATION_CAP`
