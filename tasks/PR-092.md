## Problem

No single document describes the system architecture. New contributors (human or AI) must read scattered code to understand the state machine, plugin system, auto-queue, and deployment.

## Scope

Create DESIGN.md as living architecture document. References MODEL_PROFILING.md for profiling model.

## Changes

### DESIGN.md (new)

Sections:
1. Overview: what the daemon does, one paragraph
2. State machine: IDLE -> CODING -> WATCH -> (FIX -> WATCH | MERGE) | HUNG | PAUSED | ERROR. ASCII diagram.
3. Coder plugin architecture: CoderPlugin protocol, registry, how to add a new coder
4. Auto-queue: task file headers, DAG builder, status derivation from git, QUEUE.md auto-generation
5. Rate limit system: 3 layers (proactive usage API, in-flight statusline, reactive stderr). Per-coder tracking. Auto-fallback.
6. Dashboard: HTMX polling, component structure, state publishing via Redis
7. Deployment: Docker Compose topology (daemon, web, redis). Auth volumes. Egress requirements.
8. Profiling model: link to docs/MODEL_PROFILING.md, execution_profile concept, 4 layers
9. Task file format: required header fields, validation rules

No code changes.
