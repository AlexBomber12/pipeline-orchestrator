## Problem

No single document describes the system architecture or the long-term product vision. New contributors (human or AI) must read scattered code to understand the state machine, plugin system, auto-queue, selector policy, and deployment. Equally important: architectural invariants agreed across multiple sprints (fallback as policy not chain, bandit exploration baked in, RunRecord stage field, cost-per-merged-PR as north star, etc.) live only in chat history and people's heads. Without a living document, the next coder (human or AI) can unintentionally violate these invariants and force rework. This PR captures both architecture reference and decision rationale in one file.

## Scope

Create `DESIGN.md` at repo root as living architecture document. Cover: product vision (multi-agent pipeline, current coder-only scope), north-star metric, invariants list (explicit list of "do not violate"), current architecture, decision log with dates and rationale, future agents (Planner, Reviewer) as placeholders with interface sketches. Link out to supporting docs (CODER_PRIORS.md, CODER_PRICING.md when it exists, MODEL_PROFILING.md if present).

No code changes.

## Changes

### DESIGN.md (new)

Sections, in order:

**1. Overview**. One paragraph. Pipeline Orchestrator automates software work from intent to merged PR. Current scope: Coder stage (CODING, WATCH, FIX, MERGE). Future scope: full multi-agent pipeline with Planner, Coder, Reviewer, and potentially more stages.

**2. North-star metric**. Cost per merged PR, summed across all pipeline stages. Drives routing decisions, coder selection, agent comparison. Marginal cost within subscription quota equals zero; amortized cost divides monthly subscription price by merged PRs in the month. Both metrics are tracked: marginal for routing, amortized for reporting. Pricing source: docs/CODER_PRICING.md (to be created in a future PR when per-coder pricing table is formalized).

**3. Pipeline vision**. Full stages:
- Planner: converts user intent into tasks/PR-*.md files. Not yet implemented.
- Coder: implements tasks, commits, pushes, opens PR. Implemented via CoderPlugin protocol.
- Reviewer: reviews PR, gates merge. Currently external (Codex Review). Future: internal AI reviewer alongside or replacing external.
- Merger: external (GitHub auto-merge) or internal (handlers/merge.py for conflict resolution).

Current implementation: only Coder stage has concrete agents. Planner and Reviewer are placeholders with interface sketches in section 10. AgentPlugin generalization of CoderPlugin is documented but not yet factored.

**4. Architectural invariants (do not violate)**. Explicit list. Each invariant has rationale and the PR where it was introduced.
1. CoderPlugin protocol is stable. Adding a new coder does not require changes to selector, runner, handlers, or dashboard. (PR-073, PR-077)
2. Fallback is a policy, not a chain. Selector takes state, returns ranked eligible coders. Adding Nth coder adds a priority entry, not a chain rewrite. (PR-091)
3. Bandit exploration is baked into the selector from day one. Epsilon-greedy default 0.15, configurable per daemon. Not optional: pure greedy creates data starvation for runner-up coders and prevents the router from ever discovering improvement. (PR-091)
4. Metrics capture is never optional. Every CODING/FIX/MERGE run writes a RunRecord with cost signal. Without capture, quality routing has no basis. (PR-080, PR-081, PR-082, PR-093)
5. RunRecord `stage` field distinguishes Coder runs from future Planner/Reviewer runs. Default value is `"coder"` for backward compatibility. PR-level cost aggregation sums across stages for a single (task_id, repo_name) bundle. (PR-093)
6. CoderPlugin generalizes to AgentPlugin when a second agent role arrives. Same protocol shape (run, check_auth, usage provider, rate_limit_patterns). No breaking change to coder plugins when this generalization happens.
7. Subscription vs metered pricing tracked separately. Marginal cost within subscription = 0. Amortized cost = monthly_subscription / merged_prs_per_month. Routing uses marginal, reporting uses both.
8. Multi-repo parallelism via worker pool with per-repo state isolation in Redis (keys namespaced as `repo:{name}:*`). No cross-repo state leakage.
9. CLI invocations go through `retry_transient` for network flakiness. New transient patterns added to `TRANSIENT_MARKERS`, not handled ad-hoc at call sites. (PR-054)
10. Task files (tasks/PR-*.md) are the source of truth for planned work. QUEUE.md is generated from them after PR-088 lands; manual edits to QUEUE.md are not made after that point.

**5. State machine**. ASCII diagram:
```
IDLE -> CODING -> WATCH -> (FIX -> WATCH | external merge -> IDLE) | HUNG | PAUSED | ERROR
```
Transitions: IDLE picks next eligible task from auto-queue, enters CODING. CODING runs coder plugin, opens PR, transitions to WATCH. WATCH polls PR status, transitions to FIX on review feedback or to IDLE when PR merges/closes externally. FIX runs coder plugin on review feedback, pushes, returns to WATCH. PAUSED entered on rate limit. HUNG entered on stale WATCH (no progress for N minutes). ERROR entered on unrecoverable exception. Recovery is idempotent: daemon restart resumes from last published state via Redis + GitHub introspection.

**6. Coder plugin architecture**. CoderPlugin protocol lives in `src/coder_registry.py`. Each coder is a class with: `name`, `display_name`, `models`, `run_planned_pr`, `fix_review`, `check_auth`, `create_usage_provider`, `rate_limit_patterns`. Registered in `build_coder_registry()`. Adding a new coder: create new module in `src/coders/`, register in factory, add priority entry in `docs/CODER_PRIORS.md`, add pricing entry in `docs/CODER_PRICING.md` (when created). Example walk-through: reference `src/coders/claude.py` and `src/coders/codex.py`.

**7. Selector and fallback policy**. `src/daemon/selector.py`. Three pure functions:
- `eligible_coders(ctx)`: filters out rate-limited, auth-failed, repo-disabled, pinned-out coders.
- `rank_coders(eligible, ctx)`: with probability `exploration_epsilon` returns shuffled eligible list, else sorted by `coder_priority`.
- `select_coder(ctx)`: composes both, returns top-ranked eligible coder or None.

Runner calls `select_coder` in `_get_coder`. No coder-specific branching elsewhere in runner or handlers. Injected `random.Random` for deterministic tests.

**8. Metrics capture and cost**. RunRecord schema in `src/metrics.py`. Key fields: task_id, stage (coder/planner/reviewer), profile_id (coder:model), tokens_in/out, duration_ms, fix_iterations, exit_reason, files_touched_count, languages_touched, diff_lines_added/deleted, test_file_ratio, had_merge_conflict, base_branch. PR-level cost = sum across all RunRecords with same (task_id, repo_name). Cost per merged PR per coder per task_type = bundle_cost_sum / merged_count. Pricing conversion (tokens -> dollars) uses per-coder pricing table documented in docs/CODER_PRICING.md (future).

**9. Auto-queue and task files**. Task file header parser in `src/daemon/queue_parser.py`. DAG builder in `src/daemon/dag.py`. Status derivation from git in `src/daemon/task_status.py`. Auto-queue integration in `src/daemon/handlers/idle.py`. Eligible task = dependencies satisfied and not already in flight.

**10. Rate limit system**. Three layers:
- Proactive: OAuth usage API check before CLI invocation.
- In-flight: statusline monitor during CLI execution.
- Reactive: stderr parser after CLI return.

Per-coder tracking in `state.rate_limited_coders` (set of coder names currently limited). Selector excludes rate-limited from eligible pool. Auto-recovery when rate limit window expires.

**11. Dashboard**. FastAPI + Jinja2 + HTMX. 5s polling on repo summary, 30s on CLI log. Read-only, zero AI calls. State published to Redis per-repo with key `pipeline:{name}`.

**12. Deployment**. Docker Compose with three services: daemon, web, redis. Auth volumes: `/data/auth/claude`, `/data/auth/gh`, `/data/auth/codex`. Repo checkouts: `/data/repos/{owner}__{repo}`. Config at project root `config.yml`.

**13. Future agents (placeholder)**.
- **Planner**: plugin interface mirrors CoderPlugin. Input: user intent (free text). Output: one or more `tasks/PR-*.md` files with populated headers (Branch, Type, Complexity, Depends on, Priority). Cost tracked as `stage="planner"` in RunRecord. When implemented: new module `src/planners/`, registry `build_planner_registry()`.
- **Reviewer**: plugin interface for internal AI review. Input: PR diff + recent context + AGENTS.md. Output: structured review (comments list with file/line, approve/request-changes verdict). Cost tracked as `stage="reviewer"`. May run alongside external Codex Review or replace it. When implemented: new module `src/reviewers/`, integrates with WATCH state.

AgentPlugin generalization: when second agent role arrives, extract common protocol fields from CoderPlugin (run/check_auth/usage/rate-limit) into `AgentPlugin` base, specialize per role. No breaking change to existing coder plugins.

**14. Decision log**. Chronological, one line per irreversible architectural decision with PR reference and date (use actual date when filling in).
```
- <date>: CoderPlugin protocol introduced (PR-073, PR-077). Replaced hard-coded coder branches with registry. Rationale: plugin architecture enables N coders without selector rewrites.
- <date>: Fallback as policy not chain (PR-091). Rationale: linear chains do not extend past 2 coders; Gemini/Qwen planned within 1-3 months.
- <date>: Bandit exploration baked into selector from day one (PR-091). Rationale: pure greedy starves runner-up coders, blocks future quality routing.
- <date>: RunRecord `stage` field from day one (PR-093). Rationale: adding it later requires migration on accumulated records; future Planner/Reviewer stages fit without schema change.
- <date>: Cost per merged PR as north star metric (future PR-094). Rationale: single measurable success signal aligning speed, quality, success rate, and efficiency.
- <date>: Subscription and metered costs tracked separately. Rationale: marginal cost within subscription = 0, different optimization regimes.
- <date>: Shadow mode deferred to evaluation-only role (sprint 13+). Rationale: bandit in production gives cleaner cost-per-merged-PR signal than shadow; shadow becomes evaluation sandbox for new agents (including future Planner/Reviewer).
- <date>: Pipeline vision is multi-agent (Planner + Coder + Reviewer + possibly more). Rationale: matches product direction toward end-to-end intent-to-merged-PR automation.
```

**15. Links to supporting docs**.
- `docs/CODER_PRIORS.md` (created in PR-091): per-coder initial priority values derived from public benchmarks.
- `docs/CODER_PRICING.md` (future PR): per-coder per-model pricing table, subscription amortization notes.
