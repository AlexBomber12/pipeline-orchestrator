# PR-153: Integration test infrastructure setup

Branch: pr-153-e2e-infrastructure
- Type: feature
- Complexity: high
- Depends on: none
- Priority: 1
- Coder: any

## Problem

The repo has 33 unit test files covering src/ at 100% coverage, but no end-to-end tests that validate actual daemon behavior in a running docker stack. Testing-week (Days 1 to 5, archived 2026-04-24) ran manual Playwright suites under tests-manual/auto/dayN/ that validated 4 production behaviors empirically: PR-190a coder retry counter, DATA-01 SIGKILL queue integrity, DATA-02 upload during Redis down, EXT-04 Redis down and recover. None of these are protected against regression today.

Without an integration test layer, every refactor PR after this one risks silently breaking validated behaviors, and Sprint F1.1 onwards (FINDING-2 coder pin, QUEUE.md presentation refactor, state model rewrite) is impossible to merge safely.

This PR establishes the integration test infrastructure: an isolated docker compose stack, pytest-playwright dependencies, fixture conftest, a smoke test that validates the stack starts and the dashboard responds, and gitignored test data directories. It does NOT add behavioral tests yet (PR-154 does that) and does NOT modify production scripts/ci.sh or production code under src/.

## Scope

Create a tests/e2e/ subtree with conftest fixtures and one smoke test. Add docker-compose.test.yml that spins up an isolated stack on alternative ports using the same prod-built images. Add config.test.yml with test-tuned daemon polling intervals. Add requirements-test.txt with pytest-playwright. Add a tests/e2e/README.md explaining the layout. Configure pytest to NOT collect tests/e2e/ during the unit tier so existing scripts/ci.sh continues to be fast-only.

Test stack isolation principles:
- Different network name (pipeline-orchestrator-test)
- Different container names (-test suffix)
- Different host port (18800 instead of 8800)
- Different bind mount paths (tests/e2e/data/ instead of ./data/)
- Separate Redis instance (not just a different DB on the same instance, to allow killing redis without affecting prod)
- Daemon poll_interval_sec=2 (versus prod 60) so tests run in seconds not minutes
- review_timeout_min=2, fix_idle_timeout_sec=60, planned_pr_timeout_sec=60 for fast feedback

The test stack uses the existing testbed GitHub repo at https://github.com/AlexBomber12/pipeline-orchestrator-testbed for real git operations. The testbed already exists from Days 4-5 manual runs.

`tests/e2e/conftest.py`: Python module with the following session and function fixtures.

Constants at top:
TEST_DASHBOARD_URL = "http://localhost:18800", TESTBED_SLUG = "AlexBomber12__pipeline-orchestrator-testbed", TESTBED_REPO = "AlexBomber12/pipeline-orchestrator-testbed", REPO_DIR = the absolute path of this repo (resolve from __file__ chain), TEST_DATA_DIR = REPO_DIR / "tests/e2e/data", EVIDENCE_DIR = REPO_DIR / "tests/e2e/evidence" (gitignored, screenshot output).

Session fixture `ensure_test_stack_up` (autouse=True): Runs `docker compose -f docker-compose.test.yml ps --services --filter status=running` and verifies the set contains `daemon-test`, `web-test`, `redis-test`. If missing, pytest.fail with the command needed to start the stack. After confirming compose state, polls `{TEST_DASHBOARD_URL}/api/states` for up to 30 seconds returning HTTP 200 before yielding.

Session fixture `testbed_url`: returns f"{TEST_DASHBOARD_URL}/repo/{TESTBED_SLUG}".

Session fixture `testbed_slug`: returns TESTBED_SLUG.

Session fixture `dashboard_url`: returns TEST_DASHBOARD_URL.

Function fixture `get_state(slug=TESTBED_SLUG)`: returns a callable that GETs `{TEST_DASHBOARD_URL}/api/states`, parses the JSON, finds the entry matching slug, and returns the dict, or returns None if not found. The /api/states endpoint exists at src/web/app.py:970.

Function fixture `wait_for_state(states, timeout_sec=30, slug=TESTBED_SLUG, poll_interval_sec=1.0)`: returns a callable that polls get_state until `state["state"]` is in the provided list of state strings, or raises TimeoutError after timeout. Returns the final state dict.

Function fixture `make_task_zip(pr_id, title_slug, coder="any", priority=2)`: returns a callable that creates a temporary file containing a single zipped .md task file with the given pr_id, title slug, and coder pin. The .md content follows the existing template (header with Branch, Type, Complexity, Depends on, Priority, Coder lines), Problem section with one paragraph, Scope section with one paragraph, Files to touch section with a placeholder file, Files NOT to touch section, Success criteria section with one criterion. Returns the path to the zip file. Caller is responsible for cleanup if needed.

Function fixture `upload_zip`: returns a callable that POSTs a zip file to `{TEST_DASHBOARD_URL}/repos/{TESTBED_SLUG}/upload-tasks` as multipart/form-data with the file under field name "files". Returns the requests.Response object. The endpoint exists at src/web/app.py:2137.

Function fixture `take_screenshot(name)`: returns a callable that uses Playwright's `page` fixture to capture a screenshot to EVIDENCE_DIR with timestamp prefix. Creates EVIDENCE_DIR if missing. Used for debugging failed tests; not asserted on directly.

Function fixture `reset_testbed_clean`: yields, and on teardown closes any open PRs on the testbed repo via `gh pr list -R {TESTBED_REPO} --state open --json number --jq '.[].number'` followed by `gh pr close {n} --delete-branch` for each. CRITICAL: after closing PRs, polls daemon state via get_state until `state["state"] == "IDLE"` and `state["current_pr"] is None`, with 60-second timeout. This addresses OBS-15 (Day 5 finding) where cross-test contamination occurred because tests started before daemon returned to idle. The gh commands are run via `subprocess.run` from the host (not via docker exec), since gh is available in the dev environment outside the container too.

`tests/e2e/lib/__init__.py`: empty package marker.

`tests/e2e/test_smoke.py`: a single test function `test_stack_is_up_and_testbed_configured` that uses fixtures to verify: the dashboard responds at /api/states with HTTP 200, the response contains an entry for TESTBED_SLUG, the entry has a state value among IDLE, CODING, WATCH, FIX, MERGE, ERROR, HUNG, PAUSED. The technical reason is that scripts/ci.sh is invoked by the coder inside the daemon container, which has no host docker socket and therefore cannot run docker compose for the test stack.

`docker-compose.yml`: production stack untouched. Test stack is a parallel compose file, not an override.

`config.yml`: production config untouched.

`AGENTS.md`: PR-155 updates this file. PR-153 does not.

`Dockerfile`: image is reused for both prod and test stacks. No changes needed.

`src/`: no production code changes other than the optional load_config override mentioned above.

`tests/`: existing unit tests untouched. Only the new `tests/e2e/` subtree is added.

`requirements.txt`: production requirements untouched. Test deps go in requirements-test.txt.

## Success criteria

1. `python -m pytest -q` (default invocation, equivalent to scripts/ci.sh second step) collects ONLY tests under tests/ excluding tests/e2e/. Coverage for src/ remains 100%. Test count is unchanged from before this PR plus zero (no new unit tests).

2. `pytest tests/e2e/test_smoke.py` collects exactly one test. To execute that test the test stack must be up; the test itself does not start the stack.

3. With the prod stack already running on port 8800 (existing prod from earlier setup), starting the test stack via `docker compose -f docker-compose.test.yml up -d` succeeds without port conflicts and creates 3 containers with -test suffix in their names.

4. With the test stack running, `pytest tests/e2e/test_smoke.py -v` exits 0 and reports the smoke test as PASSED.

5. With the test stack running, `curl -s http://localhost:18800/api/states` returns HTTP 200 with a JSON body containing the testbed entry. `curl -s http://localhost:18800/api/auth-status` returns HTTP 200.

6. `docker compose -f docker-compose.test.yml down -v` cleanly removes the test stack containers and volumes without affecting the prod stack on port 8800.

7. `git status` after a full test stack run shows no tracked-file modifications and no new files outside .gitignore patterns. tests/e2e/data/ and tests/e2e/evidence/ may exist locally but are gitignored.

8. `scripts/ci.sh` exits 0 unchanged in behavior. ruff check src tests passes.

9. tests/e2e/conftest.py and tests/e2e/test_smoke.py do not import from src/; they communicate with the daemon and web exclusively via HTTP and subprocess-invoked CLIs.

10. tests/e2e/README.md explains the layout and the rationale for keeping integration tier outside scripts/ci.sh.
