# PR-173: Fuzzy dedup in log_event (regex numeric counter parts)

- Type: ux
- Complexity: low
- Depends on: none
- Priority: 2
- Coder: any

## Problem

Event log spammed with near-identical messages that differ only in numeric counters. Examples:
- `PR #221 waiting (review=EYES, ci=PENDING, 1/20m)`
- `PR #221 waiting (review=EYES, ci=PENDING, 2/20m)`
- `PR #221 waiting (review=EYES, ci=PENDING, 3/20m)`
- ...
- `PR #221 waiting (review=EYES, ci=PENDING, 19/20m)`

Twenty consecutive lines, semantically identical (still waiting). Floods the dashboard log and pushes useful events off-screen.

The fix is to dedupe at log_event time using fuzzy matching: if the only difference between consecutive messages is a numeric counter pattern, treat them as one event with an updated counter.

## Scope

Backend-side dedup. `log_event(message, ...)` checks the most recent log entry for the same repo. If the new message matches the previous one modulo numeric parts, increment a counter on the existing entry instead of creating a new one. Update the timestamp.

This PR addresses backend dedup. Frontend display rendering of the dedup counter (Lucide rotate-ccw icon + count badge) covered separately in PR-174.

## Files to touch

`src/daemon/event_log.py` (or wherever log_event is implemented; search for `def log_event` in src/): add fuzzy dedup logic.

Sketch:

```python
import re

_NUMERIC_RE = re.compile(r"\d+")


def _normalize_for_dedup(message: str) -> str:
    """Replace numeric runs with placeholder for fuzzy comparison."""
    return _NUMERIC_RE.sub("#", message)


def log_event(self, message: str, ...) -> None:
    last = self._get_last_event_for_repo(repo_slug)
    if last and _normalize_for_dedup(last.message) == _normalize_for_dedup(message):
        # Same event modulo numbers — update counter and timestamp instead of new entry
        last.repeat_count = (last.repeat_count or 1) + 1
        last.message = message  # store latest version (with updated numbers)
        last.last_repeated_at = datetime.now(timezone.utc)
        self._save_event(last)
        return
    # Genuinely new event — create normally
    ...
```

`src/models.py`: add fields to event log entry dataclass / pydantic model:
- `repeat_count: int | None = None`
- `last_repeated_at: datetime | None = None`

Default None means "first occurrence, no dedup happened yet."

`tests/test_event_log.py`: add tests covering:
- Two different messages → two log entries.
- Two messages identical modulo numbers → one entry with repeat_count=2.
- Three identical-modulo-numbers in a row → one entry with repeat_count=3.
- Different message arrives after dedupe streak → new entry, repeat_count of new one is None.
- Edge case: message with no numbers is deduped on exact match.
