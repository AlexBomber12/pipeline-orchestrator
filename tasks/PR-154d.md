## Problem

PR-154b validated happy path. PR-154c covered Stop and Resume. This PR adds the third behavioral test: kill the Redis container while daemon is in IDLE, verify daemon does not crash-loop, then restart Redis and verify daemon recovers to IDLE. This protects EXT-04 (Redis down and recover) which was empirically validated during testing-week Day 5.

The test deliberately does NOT upload a task before killing Redis. Mid-task Redis kill is more complex because partial state may live in Redis. Idle-state kill is the cleaner, simpler test that validates the most important property: daemon does not panic and Docker does not restart it. A future PR can add the harder mid-task variant if regressions are found.

## Scope

Add tests/e2e/test_redis_down_and_recover.py with one test. No fixtures, no shim, no compose changes.

## Files to create

`tests/e2e/test_redis_down_and_recover.py`: one test `test_redis_kill_during_idle_does_not_crash_daemon`.

Fixtures used: dashboard_url, testbed_slug, wait_for_state, get_state, reset_testbed.

Procedure:
1. Wait for daemon IDLE: `wait_for_state(["IDLE"], timeout_sec=30)`.
2. Capture initial daemon container restart count: `subprocess.run(["docker", "inspect", "--format", "{{.RestartCount}}", "pipeline-orchestrator-daemon-test"], capture_output=True, text=True, check=True, timeout=10)`. Parse stdout as int. Store as `initial_restart_count`.
3. Stop Redis: `subprocess.run(["docker", "compose", "-f", "docker-compose.test.yml", "stop", "redis-test"], check=True, timeout=30)`.
4. Sleep 5 seconds to let the daemon notice the disconnection on its next poll cycle (poll_interval_sec=2 in test config, so 5 seconds is 2-3 cycles).
5. Probe dashboard: try `urllib.request.urlopen(f"{dashboard_url}/api/states", timeout=5)`. Three outcomes are acceptable: HTTP 200 with degraded indicator in body, HTTP 5xx (daemon survived but cannot serve due to Redis), URLError or HTTPError exception (timeout or connection reset). All three indicate "daemon may be unable to serve, but is not crashing in a loop". The unacceptable outcome is daemon being restarted by Docker, which is asserted in step 6.
6. Verify daemon container is alive: `subprocess.run(["docker", "inspect", "--format", "{{.State.Running}}", "pipeline-orchestrator-daemon-test"], ...)` returns "true".
7. Verify restart count has not changed: re-run the inspect from step 2, parse, assert equal to `initial_restart_count`. If it incremented, the daemon crashed and Docker restarted it: regression.
8. Start Redis: `subprocess.run(["docker", "compose", "-f", "docker-compose.test.yml", "start", "redis-test"], check=True, timeout=30)`.
9. Wait for Redis to be healthy: poll `subprocess.run(["docker", "compose", "-f", "docker-compose.test.yml", "ps", "--format", "json", "redis-test"], ...)` parsing JSON output and checking the Health field, OR simply sleep 10 seconds. The polling approach is more robust; the sleep approach is simpler. Either is acceptable.
10. Wait for daemon to recover: `wait_for_state(["IDLE"], timeout_sec=60)`.
11. Final: `state = get_state(); assert state is not None; assert state["state"] == "IDLE"`. Verify daemon recovered to IDLE without losing repo registration (state["slug"] == testbed_slug).

The test uses subprocess directly for docker commands (not a fixture wrapper) because docker compose invocations are infrequent here and a fixture would be overengineering for a single test.

## Success criteria

1. With the test stack up, `pytest tests/e2e/test_redis_down_and_recover.py -v` collects one test and passes within 3 minutes.

2. The test passes deterministically across at least three consecutive runs without flakiness.

3. With Redis intentionally not restarted (manual modification: comment out step 8 and run pytest), the test should fail step 10 (TimeoutError on wait_for_state IDLE) but NOT step 7 (daemon should not have restarted). This negative-test verification is performed manually during PR review by temporarily modifying the test, confirming the expected partial failure, and reverting before merging.

4. After the test passes, the redis-test container is running and healthy (the test left things tidy).

5. The daemon-test container's RestartCount is unchanged from start of test.

6. Full e2e suite `pytest tests/e2e/ -v` collects 5 tests (or 6 if shim verify is a separate test) and all pass within 12 minutes total.

7. After the test, `gh pr list -R AlexBomber12/pipeline-orchestrator-testbed --state open` returns whatever it returned before the test (Redis kill does not affect testbed PRs since no task was uploaded).

8. `python -m pytest -q` continues to exclude tests/e2e/. scripts/ci.sh exits 0.

9. ruff check src tests passes.

10. `git status` after test runs shows no tracked-file modifications.

11. The test uses explicit `timeout=` arguments on every `subprocess.run` call. No bare subprocess.run without timeout. This prevents test hangs if docker daemon itself hangs.

12. The test handles all three permissible outcomes of step 5 (HTTP 200, HTTP 5xx, exception) with proper try/except. It does NOT fail if any of these happen; it only fails if step 6 (alive check) or step 7 (no restart) fail.
