# PR-163: GitHub API rate-limit-aware polling

Branch: pr-163-github-api-rate-limit
- Type: feature
- Complexity: medium
- Depends on: PR-162
- Priority: 1
- Coder: any

## Problem

Daemon issues GitHub API requests via gh CLI without any awareness of the installation's rate limit budget. GitHub Apps have a 5000 requests/hour limit per installation. When the budget is exhausted, gh commands fail with `GraphQL: API rate limit exceeded for installation ID <id>`, and the daemon's `recover_state` (in `src/daemon/runner.py:104`) keeps retrying every 2 seconds, bombing the API further and prolonging the outage.

Observed in CI run on 2026-04-26 e2e integration: installation 127045652 hit the limit during a single full e2e test run. With test config `poll_interval_sec=2` and 6 tests, each running CODING/WATCH/FIX/MERGE cycles with full GitHub API queries (gh pr list with `statusCheckRollup`, `commits`, etc.), the budget can be exhausted in under an hour.

This is also a production risk. Production daemon at idle does ~30 API calls per hour per repo, which is comfortably under 5000. But add a few more managed repos (TaskU SaaS direction in memory #5), or hit a recovery loop where daemon polls aggressively, and production becomes vulnerable to the same exhaustion. There is no current safeguard.

The fix: read GitHub's rate-limit response headers, track remaining budget in daemon state, and adapt polling behavior when budget is low.

## Scope

Add rate-limit awareness to `src/github_client.py`. Track per-installation remaining requests and reset timestamp in Redis-backed daemon state. Adapt poll interval when budget drops below configurable thresholds. Surface budget in dashboard UI as a visibility primitive. Use the BoundedRecoveryPolicy framework from PR-162 to manage threshold actions.

This PR does NOT change the gh CLI invocation pattern (still uses `subprocess.run` via existing helpers). It does NOT replace gh with the GitHub REST API directly. It does NOT batch or cache API responses (separate optimization, not this PR). It is targeted at the rate-limit dimension only.

## Files to create

`src/daemon/github_rate_limit.py`: new module containing the budget tracker. Sketch:

```python
from __future__ import annotations

from dataclasses import dataclass
from datetime import datetime, timezone


@dataclass
class RateLimitBudget:
    installation_id: str | None  # None = personal access token
    remaining: int
    limit: int
    reset_at: datetime

    @property
    def remaining_percent(self) -> float:
        if self.limit <= 0:
            return 100.0
        return (self.remaining / self.limit) * 100.0

    @classmethod
    def from_headers(cls, headers: dict[str, str], installation_id: str | None) -> "RateLimitBudget":
        return cls(
            installation_id=installation_id,
            remaining=int(headers.get("x-ratelimit-remaining", 5000)),
            limit=int(headers.get("x-ratelimit-limit", 5000)),
            reset_at=datetime.fromtimestamp(int(headers.get("x-ratelimit-reset", 0)), tz=timezone.utc),
        )
```

The class is dataclass-frozen-ish but immutable per query. Each gh CLI response with headers produces a fresh instance. Latest is stored in Redis as `github_rate_limit_budget:{installation_id}`.

`tests/test_github_rate_limit.py`: unit tests for parsing headers, computing remaining_percent, edge cases (zero limit, missing headers, malformed values).

## Files to touch

`src/github_client.py`: extend gh CLI invocation to optionally request and parse rate-limit headers. The simplest path: use `gh api -i <endpoint>` for selected calls (returns headers + body). For the most-frequent call (gh pr list in recover_state), parse the underlying GraphQL via `gh api graphql --raw-field query='...' -i` to get headers.

Where headers are available, parse via `RateLimitBudget.from_headers` and write to Redis. Other gh commands (gh pr merge, gh pr ready, etc.) are infrequent and do NOT need header parsing on every call — periodic polling of `gh api rate_limit -i` once per minute is sufficient as a fallback budget signal.

Add a helper `get_current_rate_limit_budget()` in github_client.py that reads the latest budget from Redis, or returns a "no data" sentinel.

`src/daemon/runner.py`: in the main poll loop, before each gh API call:
1. Read current budget from Redis via the helper.
2. If `remaining_percent < threshold_critical` (e.g., 5%): pause until reset_at. Log "GitHub API budget critical, pausing until {reset_at}." Do NOT consume more requests.
3. If `remaining_percent < threshold_warning` (e.g., 20%): increase effective poll_interval by a factor (e.g., 5x). The `BoundedRecoveryPolicy` from PR-162 manages this state transition.
4. If `remaining_percent >= threshold_warning`: normal poll interval.

Threshold values come from `app_config.daemon`:
- `github_api_pause_threshold_percent: 5`
- `github_api_slowdown_threshold_percent: 20`
- `github_api_slowdown_multiplier: 5`

`config.yml`, `config.test.yml`: add the three new fields with default values. Test config gets identical values (testing rate-limit logic itself is not the concern of e2e tests; they pass through the same code path as production).

`src/web/templates/components/repo_card.html` (or equivalent dashboard component): add a visual indicator for current GitHub API budget. Sketch: a small bar showing `<remaining>/<limit>` requests, color-coded green/yellow/red based on the two thresholds. Same pattern as Claude Session % indicator already present.

If a similar Claude Session indicator does NOT exist in current code, add a minimal one for GitHub budget only — do not refactor unrelated UI in this PR.

`tests/test_github_client.py`: add tests for `get_current_rate_limit_budget()` and the gh CLI parsing extension. Mock subprocess responses with sample headers.

`tests/test_daemon_runner.py` (or wherever main loop is tested): add tests for the three threshold paths (critical pause, warning slowdown, normal). Mock the budget in Redis.

## Files NOT to touch

`src/daemon/handlers/`: handlers do not need rate-limit awareness. The poll loop in `runner.py` mediates polling rate; individual handlers see whatever budget the loop allows.

`src/models.py`: no RepoState or PRInfo schema changes. Budget is per-installation, not per-repo, and lives in its own Redis key.

`docker-compose.yml`, `docker-compose.test.yml`: untouched.

`.github/workflows/ci.yml`: untouched. Note: with this PR merged, e2e CI runs may encounter the slowdown path if budget is low, which means tests may run slower in resource-constrained reruns. This is acceptable behavior — it's the daemon being a good citizen.

`AGENTS.md`, `README.md`: untouched. Behavior is internal-to-daemon; coder/shim do not interact with rate limits.

`docs/runbooks/`: a runbook update may be useful (mentioning the budget bar and what each threshold means), but separate PR-X for docs polish.

`Dockerfile`, `scripts/`: untouched.

`requirements.txt`, `requirements-test.txt`: no new deps. Standard library datetime + dataclasses.

If the implementer finds the fix requires touching any file outside the listed sets, stop and ask in the PR description.
