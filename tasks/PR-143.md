# PR-143: Remove IDLE-only upload guard, accept uploads during any state

Branch: pr-143-upload-no-idle-guard
- Type: bugfix
- Complexity: low
- Depends on: none
- Priority: 2
- Coder: any

## Problem

The upload endpoint currently rejects requests with `422 Cannot upload while repo is {STATE}. Wait until IDLE.` whenever the repo is not in IDLE state.

This defeats the autopilot use case. The daemon is designed to run for hours or days without manual intervention. It spends most of that time in CODING, WATCH, FIX, or MERGE - not IDLE. The guard means the operator cannot queue new task files while the daemon is actively working, which is exactly when queueing more work is most natural.

The guard exists to prevent race conditions between upload commits and daemon-driven writes to tasks/. But that race is already handled architecturally:
1. Upload does NOT write to the working tree directly. It stages files in a Redis manifest.
2. The daemon drains the manifest and commits to tasks/ only during IDLE, when it is safe.
3. While the daemon is in CODING/WATCH/FIX/MERGE, it works on a feature branch. The tasks/ directory on main is not touched, so there is no race.

The guard is therefore redundant. It punishes users for the architecture working correctly.

## Scope

Remove the IDLE check in `src/web/app.py` `upload_tasks` endpoint. Upload should succeed during any repo state.

Add an informational line to the upload success message that tells the user when the files will actually be committed:
- If state is IDLE: "Accepted N files. Daemon will commit on the next poll cycle (up to 60 seconds)."
- If state is anything else: "Accepted N files. Daemon is currently {STATE}. Files will be committed when it returns to IDLE."

The status bar "Active" counter on the index page already shows that work is happening, so the user has full context.

### Edge case handling

If a file in the new upload has the same filename as an existing file in tasks/ (e.g. re-uploading PR-124.md with edits while daemon already has the original in a branch), the drain should log a clear warning about the overwrite. Do not block the upload - last write wins for the manifest - but make the log line findable so operators can spot unintentional collisions.

Do not change the Redis manifest format, the drain logic, or any other part of the flow. This is purely about lifting the guard and improving the success message.

## Files to modify

- `src/web/app.py`: remove the `if repo_state.state != PipelineState.IDLE:` block that returns the 422 error. Update the success message to include the current state-aware hint.
- `src/daemon/handlers/idle.py` (or wherever drain lives): if a file in the manifest already exists in tasks/, log a clear warning line including old and new content hash, then proceed with overwrite.
- Tests: cover the case where upload is called during CODING state and returns 200 with the correct hint message. Cover the overwrite-collision logging.

## Success criteria

- Uploading during CODING state returns 200 with message mentioning the current state.
- Uploading during IDLE returns 200 with message about the 60-second cycle.
- Files appear in tasks/ on main after the daemon returns to IDLE and drains the manifest, regardless of when the upload happened.
- Collision warning line is present in event log when a re-uploaded filename replaces an existing one.
- No `422 Cannot upload while repo is` error path remains in the endpoint.
- `scripts/ci.sh` exits 0.
- `ruff check src tests` exits 0.
- Coverage stays at 100%.
