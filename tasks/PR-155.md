# PR-155: GitHub Actions CI workflow with two-tier gate

- Type: feature
- Complexity: medium
- Depends on: PR-154
- Priority: 1
- Coder: any

## Problem

PR-153 and PR-154 deliver an integration test suite that runs locally in a docker compose stack. Without an automated runner, the suite executes only when a human remembers to run it, which means real merge gating depends on human discipline. The four production behaviors validated empirically (PR-190a retry counter, DATA-01 SIGKILL, DATA-02 upload during Redis down, EXT-04 Redis down and recover) plus everything we add in subsequent sprints are protected only as long as that discipline holds.

The chosen solution is GitHub Actions running on every pull_request and push to main. The unit tier (scripts/ci.sh) duplicates what the coder ran locally before opening the PR; this duplication is intentional, providing an authoritative result in a clean environment independent of the coder's working state. The integration tier runs the e2e suite from PR-153/154 with the mock coder shim. Both must be green for merge eligibility, alongside the existing Codex review +1 contract.

The integration job needs GitHub authentication scoped to the testbed repository (https://github.com/AlexBomber12/pipeline-orchestrator-testbed) for real git push, PR open, and comment operations. The chosen credential mechanism is a GitHub App with installation on the testbed repo, generating ephemeral 1-hour tokens per workflow run via actions/create-github-app-token. This avoids long-lived PATs that require manual rotation and provides finer-grained permissions than a user PAT.

## Scope

Add `.github/workflows/ci.yml` defining two jobs (unit and integration). Add `docs/ci-setup.md` containing the human setup steps for the GitHub App and repository secrets. Update AGENTS.md with a section codifying the new CI gate contract.

This PR does NOT include scripts/test-e2e.sh or docs/local-e2e.md. Those land in PR-155b.

## Files to create

`.github/workflows/ci.yml`: GitHub Actions workflow definition.

Workflow header: name "CI". Triggers: pull_request on branches main, push on branches main. Top-level concurrency: group equal to ${{ github.workflow }}-${{ github.ref }}, cancel-in-progress true. Top-level permissions: contents read, pull-requests write, checks write, id-token write (latter required for GitHub App token generation).

Job `unit`:
- runs-on ubuntu-latest
- steps: actions/checkout@v4, actions/setup-python@v5 with python-version 3.12 and cache pip, run pip install -r requirements.txt, run scripts/ci.sh

Job `integration`:
- runs-on ubuntu-latest
- needs: unit (only run if unit passed, saves runner minutes)
- nested concurrency block: group integration-testbed-shared, cancel-in-progress false (this serializes integration jobs across all PRs since they share one testbed repo, second PR's integration job waits for first to complete)
- steps in order:
  1. actions/checkout@v4
  2. actions/create-github-app-token@v1 with id testbed-token, app-id from secret TESTBED_APP_ID, private-key from secret TESTBED_APP_PRIVATE_KEY, owner AlexBomber12, repositories pipeline-orchestrator-testbed
  3. Set environment GH_TOKEN to the generated token output, run `gh auth status` to verify
  4. actions/setup-python@v5 with python-version 3.12 and cache pip
  5. pip install -r requirements.txt -r requirements-test.txt
  6. playwright install --with-deps chromium
  7. docker compose -f docker-compose.test.yml build (uses Dockerfile from repo)
  8. Initialize codex-auth-test volume: docker volume create pipeline-orchestrator_codex-auth-test, then docker run --rm -v pipeline-orchestrator_codex-auth-test:/target alpine chown -R 1000:1000 /target
  9. docker compose -f docker-compose.test.yml up -d --wait (--wait blocks until services healthy)
  10. Run pytest tests/e2e/ -v with GH_TOKEN env from step 2 token output. This step is the test gate.
  11. On failure (if: failure()): docker compose -f docker-compose.test.yml logs > stack-logs.txt and cat that file (logs visible in workflow output)
  12. always(): actions/upload-artifact@v4 with name e2e-evidence, path including tests/e2e/evidence/ and stack-logs.txt, retention-days 14
  13. always(): docker compose -f docker-compose.test.yml down -v (clean teardown)

Important: the GH_TOKEN passed to pytest must be the testbed-token (scoped to testbed repo only), NOT the default GITHUB_TOKEN of the workflow (scoped to pipeline-orchestrator repo where the workflow runs). The shim script in PR-154 calls gh against the testbed; it must use the testbed-scoped token to have push and PR rights.

`docs/ci-setup.md`: markdown document, target length 60 to 100 lines, written for the human operator (you, AlexBomber12).

Sections:

1. Overview: this document is read-once during initial CI setup. It explains how to provision the GitHub App that powers integration tests in GHA, and the repository secrets that reference it. Once setup is complete, the workflow runs automatically on every PR with no further human action.

2. Why GitHub App and not PAT: GitHub Apps generate ephemeral tokens per use (1-hour TTL), have fine-grained permissions per repository, and do not require periodic manual rotation. A long-lived PAT would have to be rotated quarterly and is harder to scope precisely. For a tool that will run for years and may eventually have multiple developers contributing, App-based auth is the future-proof choice.

3. Step-by-step setup. Each step has its own subsection.
   - Step A: Create the GitHub App. Navigate to https://github.com/settings/apps/new. Name it "pipeline-orchestrator-testbed-ci" or similar. Homepage URL can be the repo URL. Disable webhook (uncheck "Active"). Repository permissions: Contents Read and Write, Issues Read and Write, Pull requests Read and Write, Metadata Read (default). Subscribe to events: none. Where can this App be installed: Only on this account. Click "Create GitHub App".
   - Step B: Generate private key. After app creation, scroll to "Private keys" section, click "Generate a private key". Browser downloads a .pem file. Keep this file secure; it grants admin rights to anything the App can access.
   - Step C: Note the App ID. The number is shown at the top of the App's settings page (e.g. "App ID: 123456").
   - Step D: Install the App. Click "Install App" in left sidebar. Select your account. On the install page, choose "Only select repositories" and pick exactly `pipeline-orchestrator-testbed`. Click Install.
   - Step E: Add repo secrets to pipeline-orchestrator. Navigate to https://github.com/AlexBomber12/pipeline-orchestrator/settings/secrets/actions. Click "New repository secret". Name TESTBED_APP_ID, value the number from step C. Click Add. Click "New repository secret" again. Name TESTBED_APP_PRIVATE_KEY, value the entire contents of the .pem file from step B (including BEGIN and END lines). Click Add.
   - Step F: Verify. Trigger any workflow run (push a trivial commit or re-run a recent workflow). The integration job should successfully generate a token in the actions/create-github-app-token step. If it fails with "App not installed on repository", repeat step D.

4. Required check enforcement. The workflow file alone does not block merges; GitHub branch protection settings do. After the workflow has run successfully at least once on main, navigate to https://github.com/AlexBomber12/pipeline-orchestrator/settings/branches, edit the main branch protection rule (or create one), under "Require status checks to pass before merging" add "unit" and "integration" as required checks. Save. From this point, all PRs require both checks green plus Codex review +1 to merge.

5. Token rotation. GitHub Apps automatically generate ephemeral tokens per workflow run, no manual rotation needed. The private key (TESTBED_APP_PRIVATE_KEY secret) does not expire by default. If you ever suspect compromise, regenerate it via App settings: delete the old private key, generate a new one, update the repository secret. The old key is invalidated immediately.

6. Cost and quota. pipeline-orchestrator is a public repository, so GitHub Actions runner minutes are unlimited on free tier for public repos. The integration job consumes about 12 to 18 minutes per run; concurrency lock means at most one integration runs at a time.

## Files to touch

`AGENTS.md`: add a new section AFTER the existing "Daemon Mode" section and BEFORE "Codex Review gate (GitHub PR)". Title the section "CI gates and merge criteria".

Content (about 18 to 25 lines):

The pipeline-orchestrator repo runs CI on GitHub Actions for every pull request. The workflow has two jobs that both must pass for merge eligibility, in addition to the existing Codex review +1 requirement.

The unit job runs scripts/ci.sh in a clean environment: ruff check, pytest with 100 percent coverage requirement, no integration tests. This duplicates what the coder ran locally before opening the PR. The duplication is intentional: the local run is a pre-flight that catches issues before publishing the PR; the GHA run is the authoritative result in a clean environment, independent of any drift in the coder's working state. Coder responsibility: scripts/ci.sh exit 0 locally before opening PR. GHA validates the same.

The integration job runs the e2e suite under tests/e2e/ against an isolated docker compose stack with mock coder shim (scripts at tests/e2e/lib/coder_shim.sh). It depends on the unit job and runs only if unit passed. It serializes globally with concurrency group integration-testbed-shared because all integration jobs share one testbed repository (https://github.com/AlexBomber12/pipeline-orchestrator-testbed). Coder responsibility: NONE. The coder cannot and should not invoke docker compose from inside the daemon container that runs it; integration tests are GHA's responsibility.

Merge contract: PR is merge-eligible when all three are true. (1) Unit job green. (2) Integration job green. (3) Codex review +1 valid (non-stale per the review anchor rules). The daemon's auto-merge logic reads statusCheckRollup from the GitHub API, which aggregates all required checks; daemon already handles multi-check workflows correctly via the existing CIStatus enum logic in src/task_status.py.

Failure handling: if the unit job fails, the coder fixes it the same as scripts/ci.sh local failures (FIX REVIEW mode driven by Codex feedback or by the human). If the integration job fails, the failure is treated identically to a Codex review failure: human reviewer or coder enters FIX REVIEW mode, examines the GHA run logs and uploaded e2e-evidence artifact, fixes the issue, pushes, and waits for re-run. Integration test failures must be fixed by code change, never by disabling or skipping the test. If a test is genuinely flaky and the failure is not reproducible, the human reviewer may rerun the failed job manually via the GHA UI.

Setup prerequisite: docs/ci-setup.md describes the one-time setup of the GitHub App that powers integration tests. New deployments of pipeline-orchestrator must complete that setup before integration tests will function on their fork.

## Success criteria

1. `.github/workflows/ci.yml` is syntactically valid YAML and passes actionlint if available locally (the coder may install actionlint via go install or use the action `reviewdog/action-actionlint` informally).

2. The workflow defines exactly two jobs named `unit` and `integration` at the top level of the jobs map.

3. The unit job runs scripts/ci.sh and exits 0 on a healthy main branch.

4. The integration job has a `needs: unit` dependency. If unit fails, integration is skipped, saving runner minutes.

5. The integration job has a nested concurrency block with group `integration-testbed-shared` and cancel-in-progress false.

6. The integration job uses `actions/create-github-app-token@v1` with parameters referencing TESTBED_APP_ID and TESTBED_APP_PRIVATE_KEY secrets.

7. The integration job invokes pytest tests/e2e/ with GH_TOKEN env set to the App-generated token, NOT the default workflow GITHUB_TOKEN.

8. The integration job uploads an artifact named `e2e-evidence` containing tests/e2e/evidence/ contents and stack-logs.txt, retention 14 days, runs on always() so it captures evidence even on failure.

9. The integration job runs `docker compose -f docker-compose.test.yml down -v` on always() teardown, ensuring no leaked containers persist between runs on the same self-hosted runner (relevant if you ever migrate from ubuntu-latest hosted to self-hosted).

10. `docs/ci-setup.md` exists and contains all six steps (A through F) for human setup of the GitHub App, with explicit URLs, exact secret names, and the "Required check enforcement" subsection explaining branch protection.

11. AGENTS.md contains the new "CI gates and merge criteria" section in the correct position. The section explicitly states: coder responsibility is local scripts/ci.sh only; integration tests are GHA's responsibility; merge requires unit green AND integration green AND Codex +1.

12. PR-155 itself is permitted to fail its own integration job on first run (because secrets are not yet configured at the moment of merge). The PR description must clearly note this expected failure and instruct the human reviewer to merge the PR despite the red integration job. After merge, the human completes docs/ci-setup.md steps A through F, then enables required checks via repo settings. From the next PR onwards, the gate is enforced.

13. scripts/ci.sh exits 0. ruff check src tests passes.
