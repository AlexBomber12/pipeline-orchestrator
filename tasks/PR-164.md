# PR-164: FIX no-push deadlock circuit breaker

Branch: pr-164-fix-no-push-circuit-breaker
- Type: bugfix
- Complexity: low
- Depends on: PR-162
- Priority: 1
- Coder: any

## Problem

Daemon can get stuck in an infinite WATCH↔FIX loop when CI is red but the coder cannot identify a fix. Observed on PR #217 on 2026-04-26 (memory entry #30):

1. WATCH polls PR, sees `ci=FAILURE` and/or `review=CHANGES_REQUESTED`.
2. Transition to FIX. Daemon spawns coder with prompt "FIX REVIEW".
3. Coder reads PR state, determines "PR already green by Codex review, no FIX work required" (or similar). Exits 0 without pushing any commit.
4. Daemon logs `FIX REVIEW exited 0 but HEAD unchanged; no push, skipping @codex review`.
5. Transition back to WATCH.
6. WATCH polls PR again. CI status has not changed (because nothing was pushed). Sees `ci=FAILURE` again.
7. Goto step 2. Loop indefinitely.

The existing `fix_iteration_cap` does not catch this because it counts ALL FIX iterations, including productive ones. Once cap is hit, the PR is escalated. But cap is set to a value (e.g., 5) that allows several productive cycles. A no-push FIX produces no real progress yet still consumes a slot.

The deeper issue: the daemon has no separate counter for "consecutive FIX iterations that produced no push." That is the actual deadlock signal. A FIX iteration that DOES push is making progress (even if wrong) and should reset the no-push counter.

User had to manually restart the daemon container after 30+ minutes of no-progress loop. This must not happen.

The fix: add a counter for consecutive no-push FIX exits. After N (default 3), transition to HUNG with a clear message. PR is parked for human intervention rather than silently looping.

## Scope

Add a no-push FIX counter using the `BoundedRecoveryPolicy` framework from PR-162 (depends_on). When FIX exits without pushing, increment counter. When FIX exits WITH a push, reset counter. When counter reaches threshold, transition to HUNG. Add unit tests + e2e regression test.

This PR does NOT change FIX behavior on the productive path. It does NOT remove or modify the existing `fix_iteration_cap`. It adds an orthogonal counter that catches a different failure mode.

## Files to touch

`src/daemon/handlers/fix.py`: at the end of `handle_fix`, where the daemon currently logs `FIX REVIEW exited 0 but HEAD unchanged; no push, skipping @codex review` and transitions back to WATCH, instead:

1. Use `BoundedRecoveryPolicy` to increment a no-push counter on `current_pr` (or on `state.current_task` if PRInfo is not appropriate).
2. If counter reaches `app_config.daemon.fix_no_push_cap` (default 3), trigger threshold action: transition to HUNG, post explanatory comment on PR, log clear event.
3. If counter below threshold, return to WATCH as today.

When FIX produces a push (`HEAD changed`), explicitly reset the no-push counter via `policy.reset(current_pr)`. This goes alongside the existing `current_pr.fix_iteration_count` increment. The two counters are independent.

The HUNG transition message: `"FIX deadlock: {N} consecutive no-push FIX cycles on PR #{number}. Coder unable to identify actionable fix. Manual review required."`

`src/models.py`: add new field `no_push_fix_count: int = 0` to `PRInfo` (or wherever `fix_iteration_count` already lives). Default 0. Reset on PR creation.

`config.yml`, `config.test.yml`: add `fix_no_push_cap: 3` to `daemon` block. Same value in both configs.

`tests/test_fix_handler.py`: add tests covering:
- Three consecutive no-push FIX cycles → state HUNG, counter reset, expected log message
- Two no-push cycles followed by a productive push → counter reset, no HUNG
- One no-push cycle, then productive, then no-push again → counter resets between, no HUNG
- Counter is independent of `fix_iteration_count` (one can advance while the other resets)

`tests/e2e/test_fix_no_push_deadlock.py` (new): end-to-end test that creates a PR scenario where shim returns exit 0 without push N times in a row, daemon transitions to HUNG with the expected message. Use the existing shim infrastructure with a mode that exits 0 without writing.

## Files NOT to touch

`src/github_client.py`: untouched. Same gh CLI calls, no API changes.

`src/daemon/preflight.py`: untouched. Existing dirty-tree policy already refactored under PR-162.

`AGENTS.md`, `README.md`, `docs/`: untouched. Behavior is internal failure-mode handling. Optional follow-up runbook update can document the HUNG condition; not blocker for this PR.

`Dockerfile`, `scripts/`: untouched.
