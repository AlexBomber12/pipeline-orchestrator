# PR-135: Redis Pub/Sub event bus and SSE endpoint infrastructure

- Type: feature
- Complexity: medium
- Depends on: none
- Priority: 2
- Coder: any

## Problem

The dashboard currently relies on HTMX polling to refresh state. Polling is wasteful on IDLE repos and noticeably laggy on state changes (up to N seconds before the user sees a transition). The plan is to migrate to Server-Sent Events with Redis Pub/Sub as the event bus, so the daemon can push updates to all connected clients immediately.

This PR lays the infrastructure foundation: a Redis Pub/Sub publisher helper in the daemon, an SSE endpoint in FastAPI that subscribes to the channel, and the hooks for future event types. It does NOT wire any specific UI element to SSE yet - that comes in subsequent PRs (starting with PR-136 for sprint progress).

## Scope

### Publisher side

Add `src/events/publisher.py`:
```
async def publish_repo_event(repo_name: str, event_type: str, payload: dict) -> None:
    """Publish an event to the Redis channel for this repo.
    
    Also appends to a Redis List for persistence (last 50 events retained)
    so clients that reconnect can replay recent history.
    """
```

Use Redis channel name pattern: `repo-events:{repo_name}`.

Use Redis List name pattern: `repo-events-history:{repo_name}` with LTRIM to last 50.

Event payload shape:
```
{
  "type": "progress_updated" | "state_changed" | "event_log_appended" | ...,
  "repo": "AlexBomber12__pipeline-orchestrator",
  "data": { ... },
  "timestamp": "2026-04-20T14:30:00Z"
}
```

### Subscriber side (SSE endpoint)

Add `GET /api/repos/{repo_name}/events` endpoint in `src/web/app.py`:
1. On connect, reply with SSE headers (`Content-Type: text/event-stream`, `Cache-Control: no-cache`, `X-Accel-Buffering: no` for Nginx).
2. First action: replay last 20 events from `repo-events-history:{repo_name}` so the client has context after reconnect.
3. Then subscribe to Redis channel `repo-events:{repo_name}` and forward each message as an SSE event.
4. Send keep-alive comment `:keepalive\n\n` every 15 seconds so proxies do not time out the connection.
5. Clean up subscriber on disconnect (client-close or server-shutdown). No dangling Redis connections.

### Test helpers

Expose `publish_repo_event()` as a sync-testable helper. Add tests with a fakeredis fixture covering: publish increments list, subscribe receives published events, history replay on reconnect, connection cleanup on disconnect.

## Success criteria

- `await publish_repo_event("repo", "test_event", {"foo": "bar"})` appends to history list and publishes to channel.
- `curl -N http://localhost:8000/api/repos/<n>/events` opens an event stream that (a) sends last 20 history events immediately, (b) sends keepalive every 15 seconds, (c) forwards any new publish within a few milliseconds.
- Stopping `curl` closes the Redis subscriber cleanly (verified via `redis-cli CLIENT LIST`).
- No event bus client remains when the last SSE consumer disconnects.
- `scripts/ci.sh` exits 0.
- `ruff check src tests` exits 0.
- Coverage at 100% for the new files.
