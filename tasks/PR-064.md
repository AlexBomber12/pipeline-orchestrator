# PR-064: In-flight rate limit monitor via statusline hook

Branch: pr-064-statusline-inflight-monitor

- Type: feature
- Complexity: high

## Problem

Proactive check in PR-063 catches usage breaches before a Claude CLI call starts. But a single long CODING run can take 5-15 minutes and may cross the session threshold mid-flight. Without an in-flight monitor:
- A CODING that starts at 85% session may end at 105%, wasting tokens and flipping state to ERROR via 429
- Weekly limit can cross threshold during a single Opus run on a large task
- The daemon has no way to abort a runaway CLI call without waiting for it to finish or to time out

Claude CLI >= 1.2.80 supports a `statusLine` hook in `~/.claude/settings.json`. On every update during a run, Claude CLI pipes a JSON object to stdin of the configured command. That JSON includes `rate_limits.five_hour.used_percentage` and `rate_limits.seven_day.used_percentage` live. This is a documented escape hatch for exactly this problem.

## Root cause

Statusline hooks are the CLI's own real-time telemetry channel. The daemon currently does not register one, so it has zero visibility into in-flight usage. Hooking into it gives the daemon a cheap, documented way to abort a CLI call when limits are breached mid-run.

## Scope

Add a statusline hook script that:
1. Reads JSON from stdin on every update
2. Compares `rate_limits.five_hour.used_percentage` and `rate_limits.seven_day.used_percentage` against the daemon's configured thresholds
3. On breach, writes a breach marker file the daemon monitors
4. Outputs a short statusline string (the hook's primary contract) so Claude CLI does not error out

Add a monitor task in the daemon that:
1. Watches the breach marker file during an active Claude CLI call
2. On breach, kills the subprocess and sets state to PAUSED with `rate_limited_until = resets_at`
3. Advances the feedback baseline so the partial run is not treated as a push

Register the hook in `~/.claude/settings.json` inside the auth volume on daemon startup, so every Claude CLI subprocess inherits it.

Depends on PR-063 (shared OAuth usage infrastructure, PAUSED state, `UsageSnapshot` dataclass). Depends on sprint 8 foundation PRs that introduce instrumentation and the error classification taxonomy, since in-flight kills must be distinguished from OOM kills, timeout kills, and operator-initiated kills.

## Changes

### scripts/statusline_hook.py (new)

Python script executed by Claude CLI on every statusline update. Fast, no network calls, no dependencies beyond stdlib.

```python
from __future__ import annotations

import json
import os
import sys
import time
from pathlib import Path


def main() -> int:
    raw = sys.stdin.read()
    if not raw:
        sys.stdout.write("pipeline-orchestrator")
        return 0
    try:
        data = json.loads(raw)
    except json.JSONDecodeError:
        sys.stdout.write("pipeline-orchestrator")
        return 0

    breach_dir = Path(os.environ.get("PIPELINE_BREACH_DIR", "/tmp/pipeline-breach"))
    breach_dir.mkdir(parents=True, exist_ok=True)

    session_threshold = int(os.environ.get("PIPELINE_SESSION_THRESHOLD", "95"))
    weekly_threshold = int(os.environ.get("PIPELINE_WEEKLY_THRESHOLD", "100"))

    rate_limits = data.get("rate_limits") or {}
    five_hour = rate_limits.get("five_hour") or {}
    seven_day = rate_limits.get("seven_day") or {}
    session_pct = int(five_hour.get("used_percentage") or 0)
    weekly_pct = int(seven_day.get("used_percentage") or 0)

    breach_type = None
    resets_at = 0
    if session_pct >= session_threshold:
        breach_type = "session"
        resets_at = int(five_hour.get("resets_at") or 0)
    elif weekly_pct >= weekly_threshold:
        breach_type = "weekly"
        resets_at = int(seven_day.get("resets_at") or 0)

    pid = os.environ.get("PIPELINE_CLAUDE_PID", "")
    if breach_type and pid:
        marker = breach_dir / f"{pid}.breach"
        marker.write_text(json.dumps({
            "type": breach_type,
            "resets_at": resets_at,
            "session_pct": session_pct,
            "weekly_pct": weekly_pct,
            "detected_at": time.time(),
        }))

    sys.stdout.write(
        f"orchestrator s{session_pct}% w{weekly_pct}%"
    )
    return 0


if __name__ == "__main__":
    sys.exit(main())
```

The hook is side-effect-minimal: writes one small JSON file per breach, keyed by PID so multiple parallel runs (sprint 10 compete mode) do not collide.

### src/claude_cli.py

Pass the subprocess PID via env var to the hook so the hook writes a PID-keyed marker. Pass thresholds too so the hook does not need to read config.yml.

In `run_claude_async`:

```python
env = {
    **os.environ,
    "NODE_OPTIONS": _build_node_options(),
    "PIPELINE_BREACH_DIR": breach_dir,
    "PIPELINE_SESSION_THRESHOLD": str(session_threshold),
    "PIPELINE_WEEKLY_THRESHOLD": str(weekly_threshold),

proc = await asyncio.create_subprocess_exec(...)
env["PIPELINE_CLAUDE_PID"] = str(proc.pid)
```

Note: env must be set before subprocess creation, but PID is known only after. Workaround: write PID to a small file that the hook reads, or accept that first few hook invocations have empty PID (they will skip the breach check, which is fine because the first 1-2 updates are very early in the run and usage has not shifted yet).

Cleaner: export `PIPELINE_CLAUDE_PID` to `${PPID}` and have the hook read its own parent PID. Parent of the hook script IS the Claude CLI subprocess. This avoids the chicken-and-egg. Confirm on a real run which PID the hook sees.

Return breach marker path from `run_claude_async` so the caller can check it after the run too (belt and suspenders).

### src/daemon/runner.py

Add breach monitor task alongside heartbeat in `handle_coding` and `handle_fix`:

```python
async def _monitor_inflight_breach(
    self,
    breach_dir: Path,
    pid_getter: Callable[[], int | None],
    claude_task: asyncio.Task,
) -> None:
    while not claude_task.done():
        pid = pid_getter()
        if pid is not None:
            marker = breach_dir / f"{pid}.breach"
            if marker.is_file():
                try:
                    data = json.loads(marker.read_text())
                except (OSError, json.JSONDecodeError):
                    await asyncio.sleep(1)
                    continue
                self.state.rate_limited_until = datetime.fromtimestamp(
                    data["resets_at"], tz=timezone.utc
                )
                self.state.state = PipelineState.PAUSED
                self.log_event(
                    f"In-flight breach: {data['type']} at "
                    f"{data.get('session_pct' if data['type'] == 'session' else 'weekly_pct')}%, "
                    f"killing Claude CLI"
                )
                claude_task.cancel()
                return
        await asyncio.sleep(2)
```

Wire it into `handle_coding` and `handle_fix` next to the existing heartbeat task. When the monitor cancels the claude task, the existing cancellation path handles cleanup.

### src/daemon/main.py (or wherever daemon startup lives)

On startup, write `~/.claude/settings.json` in the auth volume with a statusLine block pointing to the hook script. Idempotent: merge with existing settings, do not clobber other keys.

```python
def _install_statusline_hook(claude_config_dir: str, hook_path: str) -> None:
    settings_path = Path(claude_config_dir) / "settings.json"
    try:
        existing = json.loads(settings_path.read_text()) if settings_path.is_file() else {}
    except (OSError, json.JSONDecodeError):
        existing = {}
    existing.setdefault("statusLine", {})
    existing["statusLine"] = {
        "type": "command",
        "command": f"python3 {hook_path}",
        "padding": 0,
    }
    settings_path.parent.mkdir(parents=True, exist_ok=True)
    settings_path.write_text(json.dumps(existing, indent=2))
```

Operator override: if `claude_config_dir/settings.json` already has a statusLine set to something else, log a warning but do not overwrite. User may have custom ccusage statusline configured. Add a config flag `daemon.install_statusline_hook: bool = True` for opt-out.

### Breach marker cleanup

Stale markers accumulate in `/tmp/pipeline-breach/`. Add cleanup:
- On daemon startup: remove all files
- After each run, whether it breached or not, the caller removes the PID marker
- Periodic sweep of files older than 1 hour in case of orphans

## Risks

- Hook PID resolution is fragile. The hook needs to know which Claude CLI subprocess it is reporting for. `${PPID}` works if the hook runs directly as a child of Claude CLI, but if Claude CLI shells out via `/bin/sh -c "python3 hook.py"` then PPID is the shell, not Claude. Mitigation: test on real CLI, document the resolved PID chain. Fallback: the daemon already has the subprocess PID from asyncio, it can poll the single breach marker under that PID pattern.
- Statusline hooks are per-user config in `~/.claude/settings.json`. In docker the auth volume maps to `/data/auth/claude`, HOME may not match. The daemon installs to the volume path explicitly (not `~`).
- Operator may already use ccusage statusline. The install step respects that and logs a warning instead of clobbering. Document how to opt out.
- If the hook errors or hangs, Claude CLI may disable it or stall. Hook must be fast (<100ms) and never raise. Stdlib only, no network, no file locks.
- Killing the Claude CLI mid-run may leave the working tree dirty. Existing recovery path (`_recover_from_unclean_tree`) handles this, but verify under PR-064-specific test scenarios.
- The hook writes to `/tmp/pipeline-breach/`. This path is container-local; for a multi-container setup in sprint 10 (parallel runners), the path must be a shared volume or per-runner container-local. Keep container-local for now, revisit in sprint 10.

- No regression in runtime or memory footprint of normal CODING or FIX runs when no breach occurs
