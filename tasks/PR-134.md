# PR-134: Consistent hover states on interactive elements

- Complexity: low
- Depends on: none
- Priority: 2
- Coder: any

## Problem

Many interactive elements in the dashboard (buttons, icon controls, form triggers, nav items) have inconsistent or missing hover states. Some have `hover:text-accent`, some have `hover:border-accent/30`, some have nothing. On touch devices hover is irrelevant, but on desktop discoverability suffers when a clickable element looks identical to a static one.

This PR standardizes hover treatment so every clickable element provides unambiguous feedback.

## Scope

Define two reusable hover patterns via Tailwind utility groups or CSS classes:

1. **Primary-interactive** (buttons, main actions): border color shift + slight background change on hover. Example: `hover:border-accent/50 hover:bg-white/5 transition-colors duration-150`.
2. **Secondary-interactive** (icons, text links, subtle controls): text color shift only. Example: `hover:text-accent transition-colors duration-150`.

Apply consistently across:
- Upload tasks button/label on repo card
- Play / Pause / Stop icons from PR-128 (secondary-interactive pattern)
- Theme toggle from PR-133 (secondary-interactive)
- Settings page save buttons and form controls (primary-interactive)
- Navigation links in top bar (secondary-interactive)
- Repo card hover (already exists as `hover:border-white/10` - keep, may just verify)

Add `cursor-pointer` where it is missing on interactive elements that are not natively clickable (e.g. `<label>` wrapping a file input).

Do NOT change click behavior or add new interactions. Purely visual feedback improvements.
