# PR-156: GHA workflow DNS race fix and PATH override hardening

Branch: pr-156-gha-dns-and-path-fix
- Type: bugfix
- Complexity: low
- Depends on: none
- Priority: 1
- Coder: any

## Problem

The first GitHub Actions integration run after PR-155a/b merged failed with three independent root causes documented in stack-logs.txt artifact from run YYYY-MM-DD:

1. **DNS race between Docker Compose service_healthy and DNS propagation.** The daemon-test container starts after `redis-test` reports healthy via depends_on condition, but Docker bridge network DNS records for `redis-test` are not yet propagated. Daemon's first cycle attempts to connect to `redis://redis-test:6379/0` and gets `socket.gaierror: [Errno -3] Temporary failure in name resolution`. The error fires three times over 4 seconds (18:54:38, 18:54:40, 18:54:42) before DNS finally works. By that time daemon has logged ERROR three times.

2. **PATH override missing in docker-compose.test.yml.** The PR-154a task file specified `PATH: /usr/local/bin:/home/runner/.npm-global/bin:/usr/bin:/bin` env var on web-test and daemon-test services, ensuring shim mounts at /usr/local/bin/claude and /usr/local/bin/codex take precedence over real binaries from npm-global. The merged docker-compose.test.yml omits this env var. By coincidence the shim does activate (likely because base image PATH already happens to start with /usr/local/bin), but this is fragile: any Dockerfile change to PATH ordering would silently switch the test stack to using real claude/codex CLIs that would burn quota and call real APIs.

3. **shim "no DOING task in QUEUE.md, exiting 0".** Daemon picked task PR-301 from a stale tasks/QUEUE.md on the testbed main branch (artifact of manual day5 testing-week run), but the shim looking at the same QUEUE.md somehow could not parse the DOING entry. This was resolved out-of-band by manually cleaning tasks/ from testbed main on 2026-04-25, and a separate PR-157 adds an automated session-start cleanup fixture so this state cannot accumulate again. PR-156 does NOT address #3 directly; PR-157 does.

This PR addresses #1 and #2.

## Scope

Add a sleep step in the GHA integration job after `docker compose up -d --wait` to give Docker DNS propagation time. Add explicit verification step that `which claude` returns the shim path before pytest runs. Add the missing PATH env var to both services in docker-compose.test.yml.

These are minimal, surgical changes. No production code touched.

## Files to touch

`.github/workflows/ci.yml`: insert two new steps in the integration job, both BEFORE the "Run e2e tests" step.

After the existing "Bring up test stack" step (which runs `docker compose -f docker-compose.test.yml up -d --wait`), insert:

```yaml
      - name: Wait for Docker DNS propagation
        run: sleep 20

      - name: Verify shim is mounted in daemon-test
        run: |
          actual=$(docker compose -f docker-compose.test.yml exec -T daemon-test which claude)
          if [ "$actual" != "/usr/local/bin/claude" ]; then
            echo "ERROR: shim not mounted; daemon-test which claude returned: $actual" >&2
            echo "Expected /usr/local/bin/claude. Check docker-compose.test.yml volume mounts and PATH env." >&2
            exit 1
          fi
          echo "OK: shim mounted at $actual"
```

The 20-second sleep is empirical: logs show DNS recovered within 7 seconds in the failing run, but a 3x safety margin handles slower runner instances. The sleep is in the integration job only; local development via scripts/test-e2e.sh does not need it because Docker DNS propagation on local hosts is faster (no shared kernel network namespace contention).

The verification step fails fast if PATH ordering is wrong. Without it, a misconfigured stack would silently invoke real claude/codex CLIs and burn quota; with it, the integration job stops within 30 seconds with a clear error pointing at the misconfiguration.

`docker-compose.test.yml`: add a PATH env var to BOTH `web-test` and `daemon-test` services.

In each service's `environment:` block, add a new line:
```yaml
      PATH: /usr/local/bin:/home/runner/.npm-global/bin:/usr/bin:/bin:/usr/local/sbin:/usr/sbin:/sbin
```

Place it alphabetically: between HOME and PO_CONFIG_PATH, or wherever fits the existing convention. The exact placement does not affect correctness.

The PATH order matters. /usr/local/bin (where the shim is mounted) MUST come first. /home/runner/.npm-global/bin (where the real claude/codex from `npm install -g` live) is included second so that if the shim ever needs to invoke the real binary as fallback, it can. The remaining /usr/bin:/bin:/usr/local/sbin:/usr/sbin:/sbin entries cover standard Linux tools (git, gh, awk, sed, etc.).

## Files NOT to touch

`docker-compose.yml`: production stack. PATH issue does not affect production because production has no shim mounts and PATH ordering does not matter there.

`config.yml`, `config.test.yml`: untouched.

`Dockerfile`: tempting to also pin PATH at build time via ENV directive, but that risks affecting production stack behavior. PATH-via-compose env is the targeted fix; Dockerfile-level pinning is a separate larger change for a future PR.

`src/`: no production code changes. PR-157 adds a Python fixture, NOT this PR.

`scripts/ci.sh`: untouched.

`scripts/init-test-volume.sh`: untouched.

`scripts/test-e2e.sh`: untouched. Local invocation does not need the GHA-only sleep because local Docker DNS is faster.

`tests/`: no test code changes. The shim verification is in the GHA workflow as a step, not as a pytest fixture.

`requirements.txt`, `requirements-test.txt`: untouched.

`AGENTS.md`, `README.md`, `docs/`: untouched.

If the implementer finds the fix requires touching any file outside the listed sets, stop and ask in the PR description.
