# PR-137: Settings page logical grouping and one-line hints

Branch: pr-137-settings-clarity
- Type: feature
- Complexity: low
- Depends on: none
- Priority: 2
- Coder: any

## Problem

The settings page has many fields mixed together without clear grouping. Some fields have long multi-sentence hint text that users skip. Others have no hint at all. Navigation is a single scroll through an uncategorized form.

The principle: logical grouping with one concise hint per field. Long docs belong in external reference material.

## Scope

Restructure `settings_daemon.html` and adjacent partials into named sections:
- **Timeouts** - PLANNED PR timeout, FIX timeout, review timeout
- **Rate limits** - session pause percent, weekly pause percent
- **Self-heal** - diagnose coder setting, diagnose enable flag, escalate thresholds
- **Coders** - existing coder settings section, kept as-is
- **Auth** - existing auth status section, kept as-is

Each section has:
- A short section header (bold or slightly larger text)
- Optional 1-sentence context line ("What this section controls")
- Fields with ONE-LINE hints only (currently many have multi-sentence hints - collapse to one line each)

Trim existing hint text to one concise line per field. Examples of current text to trim:
- `"Max time Claude gets before being killed."` -> `"Seconds before the coder subprocess is killed if no activity."` (and from PR-129 swap Claude -> coder)
- `"Kills Claude during FIX if no push. Resets on each push."` -> `"FIX time budget, reset on each successful push."`
- `"Let Claude analyze failures before falling back to default recovery."` -> `"Use the coder to diagnose ERROR before generic recovery kicks in."`

Do not remove any fields. Do not change form semantics or validation. Purely reorganize + rewrite hints.

Field ordering within each section by user frequency of adjustment (most-touched first).

## Files to modify

- `src/web/templates/components/settings_daemon.html` - primary target.
- Potentially extract section helper partials `_settings_section.html` for reusable header markup.
- Update any CSS selectors that may have been tied to original DOM order (search for `#daemon-form input[name=...]`).
- Tests: render test verifying all original fields still present and submit path unchanged.

## Success criteria

- Settings page shows clearly delineated sections with headers.
- Every field has exactly one-line hint, no multi-sentence descriptions.
- All original form fields are still submitted correctly (no regression in the settings save flow).
- Visual verification that layout is calmer and more scannable.
- `scripts/ci.sh` exits 0.
- `ruff check src tests` exits 0.
