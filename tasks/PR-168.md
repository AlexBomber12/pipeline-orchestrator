# PR-168: Pin npm versions for claude-code and codex CLIs

- Type: config
- Complexity: low
- Depends on: none
- Priority: 2
- Coder: any

## Problem

`Dockerfile` line 28 and 31 install `@anthropic-ai/claude-code` and `@openai/codex` without version pinning:

```
&& npm install -g @anthropic-ai/claude-code
...
&& npm i -g --prefix /home/runner/.npm-global @openai/codex
```

This means every Docker rebuild pulls the latest version of each CLI. If either upstream releases a breaking change (CLI flags rename, output format changes, stdout format changes, behavior shifts), the daemon can break silently on next rebuild without any code change on our side.

This is a real production risk. Particularly for OAuth-managed CLIs where authentication flow can change incompatibly between versions. We do not have a CI safety net that catches "upstream changed an undocumented behavior" — those failures only surface in production.

The fix is to pin both CLIs to specific known-working versions. Document the upgrade procedure: a deliberate Aleksei-driven action, not a "happens automatically on rebuild" event.

## Scope

Modify Dockerfile to install specific versions of both CLIs. Add a comment documenting the upgrade procedure. Optionally, add a CI check that warns if pinned versions diverge significantly from latest.

This PR does NOT remove the ability to upgrade. It does NOT add automatic version-bumping. It just ensures that "what gets installed" is deterministic and intentional rather than accidental.

## Files to touch

`Dockerfile`: change the npm install lines to pin versions. Sketch:

```dockerfile
# Pinned to known-working versions. Upgrade procedure:
#   1. Test new version locally via separate Docker build.
#   2. Update version below.
#   3. Open a PR-NNN: "bump claude-code to X.Y.Z" with rationale.
#   4. Verify e2e tests pass.
#   5. Merge after Aleksei review.
ARG CLAUDE_CODE_VERSION=2.1.0
ARG CODEX_VERSION=0.122.0

RUN npm install -g @anthropic-ai/claude-code@${CLAUDE_CODE_VERSION} \
    && npm i -g --prefix /home/runner/.npm-global @openai/codex@${CODEX_VERSION}
```

(Use ARG so the version is overridable at build time without editing Dockerfile, but defaulted to a known value committed to repo. Implementer should pick the actual current versions: check `npm view @anthropic-ai/claude-code version` and `npm view @openai/codex version` and use those values. Or use the versions currently in production on AI-Server — implementer should query those if accessible.)

For Codex CLI: per memory entry #13, current version on Aleksei's setup is `0.122`. So `CODEX_VERSION=0.122.0` (or whatever exact tag npm has).

For Claude Code: implementer should determine current production version. If unable, choose `latest` and explicitly document in the PR description why it's not pinned with rationale.

`docs/runbooks/upgrade-cli-versions.md` (new): brief runbook explaining:

- When to upgrade (security patches, breaking-fix, intentional feature adoption)
- How to upgrade (Docker rebuild with `--build-arg CLAUDE_CODE_VERSION=X` for testing, then commit Dockerfile change)
- How to roll back (revert Dockerfile, rebuild)
- What signals to monitor after upgrade (OAuth still works, prompts still produce expected outputs, e2e tests still pass)

If `docs/runbooks/` directory does not exist, create it with this single file. Skill agnostic — short, factual, ~30 lines.

`README.md` or `AGENTS.md` (whichever has the "Local development setup" section): brief mention "CLI versions are pinned in Dockerfile. See docs/runbooks/upgrade-cli-versions.md to upgrade."

`tests/test_dockerfile_pins.py` (or `scripts/check_dockerfile_pins.sh`): optional but recommended — a small script or test that verifies the Dockerfile contains version pins (not bare `@latest` or unversioned). Can be a shell-script invoked from `scripts/ci.sh` or a python test using regex on the Dockerfile content.

If the test is python, place under `tests/` and verify both CLIs have version pinning syntax via regex matching.
