# PR-161: Daemon git fetch uses --prune to drop stale remote-tracking refs

Branch: pr-161-git-fetch-prune
- Type: bugfix
- Complexity: low
- Depends on: none
- Priority: 1
- Coder: any

## Problem

The daemon's `git fetch origin {branch}` calls in `src/daemon/repo_ops.py` (lines 84, 167) and elsewhere do NOT pass `--prune`. As a result, when a remote branch is deleted (for example via `gh pr close --delete-branch` from a test cleanup, or any external branch deletion), the local clone in `/data/repos/{slug}/.git/refs/remotes/origin/{branch}` continues to point at the pre-deletion SHA. The remote-tracking ref becomes stale.

This is benign for read-only operations but breaks `git push --force-with-lease`. The lease compares the local remote-tracking ref to the actual remote head; if remote has been deleted, the comparison sees "remote head is none" while local says "remote head is SHA-X" and refuses with `(stale info)`. The coder process exits non-zero, daemon transitions to ERROR, subsequent tests cascade to ERROR.

Diagnosis confirmed by Claude in PR-159 self-review (PR #217 commit 6c5fdfc), reproduced in CI run 24961823433 logs:

```
The container's local origin/pr-1777222119-e2e-sigkill-recovery remote-tracking
ref still points to the pre-deletion SHA because nothing runs git fetch --prune.
git push --force-with-lease then refuses with (stale info), codex shim exits 1,
daemon → ERROR.
```

This is a real production bug. Even outside e2e tests, any external branch deletion (user closes a PR via GitHub web UI, repo admin force-cleans branches, etc.) will produce the same stuck state.

## Scope

Modify the daemon's git fetch invocations to pass `--prune`. This drops stale remote-tracking refs for branches that no longer exist on origin. Add unit tests covering the prune behavior.

This PR does NOT change push logic. It does NOT change branch creation or checkout logic. It does NOT remove any retry behavior. It is a single-flag addition to existing git fetch commands.

## Files to touch

`src/daemon/repo_ops.py`: find every `git_ops._git(self.repo_path, "fetch", "origin", ...)` invocation in the file. Add `--prune` flag immediately after `"fetch"`. Two known sites: line 84 area (fetch `self.repo_config.branch`) and line 167 area (fetch with branch argument).

The `_git` helper in `src/git_ops.py` accepts variadic args, so `_git(repo_path, "fetch", "--prune", "origin", branch)` constructs `git -C repo_path fetch --prune origin branch`. This deletes any local `origin/X` ref where remote no longer has `X` while still fetching `branch`.

Other files that may invoke `git fetch`: scan with `grep -rn "fetch.*origin\|fetch\", \"origin" src/` to find them all. Found at:
- `src/daemon/handlers/merge.py:52` — `git fetch origin {base} {pr_branch}`
- `src/daemon/handlers/merge.py:250` — `git fetch origin {base}`
- `src/daemon/handlers/fix.py:176` — `git fetch origin {branch}`
- `src/daemon/repo_ops.py:84` — `git fetch origin {branch}`
- `src/daemon/repo_ops.py:167` — `git fetch origin {branch}`

ALL of these need the `--prune` flag added. Five sites total.

The `--prune` flag is safe to add globally — it only removes refs that no longer exist on origin. It does not affect refs that DO exist on origin. There is no scenario in which `--prune` damages local state.

`tests/test_repo_ops.py` (and `tests/test_merge_handler.py`, `tests/test_fix_handler.py` if they exist for the merge/fix handlers): add or extend tests that mock `git_ops._git` and assert the call args contain `"--prune"` for fetch operations.

If existing tests for these modules do not assert on argument lists (and just mock with `Mock()` returning success), extend them to capture call_args and verify the `--prune` flag is present.
