# PR-148: Separate "Commits" metric matching GitHub Commits count

Branch: pr-148-commits-match-gh
- Type: ux
- Complexity: low
- Depends on: none
- Priority: 2
- Coder: any

## Problem

The repo detail page shows `Pushes: N` driven by `PRInfo.push_count`. The field has mixed semantics:

1. Initialized from `len(commits)` (GitHub commits count) in `src/github_client.py:190` when state is refreshed from GitHub.
2. Incremented daemon-side in `src/daemon/handlers/fix.py:368` and `src/daemon/handlers/error.py:271` on each push event the daemon detects.

After a few iterations the number diverges from GitHub's UI. On PR-138 during the recent sprint, dashboard showed `Pushes: 10` while `gh pr view 138 --json commits | jq '.commits | length'` returned `1` (after squash-style force-push history).

Users looking at both surfaces cannot reconcile the numbers. Decision: display a separate metric that ALWAYS matches GitHub Commits count. Do not modify `push_count` in this PR - it is still used by FIX iteration logic and will be deprecated later.

## Scope

### New field

Add `commits_count: int = 0` to `PRInfo` in `src/models.py`.

### Population

In `src/github_client.py` around line 190 where `push_count=len(commits)` is set, also set `commits_count=len(commits)`. This is the authoritative source. No daemon-side increments - commits_count always reflects GitHub state at last refresh.

### Display

In `src/web/templates/components/repo_summary.html` (and any other template showing the current PR card), add a new row labelled `Commits` reading `repo.current_pr.commits_count`. Place immediately before or after the existing `Pushes` row.

Do NOT remove the Pushes row in this PR. Both metrics coexist during transition. Future PR will evaluate whether Pushes row still adds diagnostic value once Commits is live.
