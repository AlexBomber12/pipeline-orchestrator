# PR-169: Hot reload config.yml on file change

Branch: pr-169-config-hot-reload-file-watch
- Type: feature
- Complexity: low
- Depends on: none
- Priority: 1
- Coder: any

## Problem

Daemon already has a hot reload mechanism (`src/daemon/runner.py:316` calls `load_config()` and updates `self.app_config` after reading a `dirty_key` flag from Redis). But the dirty_key flag is only written by the UI Settings save endpoint. When a user manually edits `config.yml` on disk (via vim, scp, docker cp, or any other path), the daemon does NOT detect the change. Daemon continues running with stale in-memory config.

Verified empirically on 2026-04-26: Aleksei edited `planned_pr_timeout_sec` from 900 to 2400 in config.yml on production AI-Server. Daemon did NOT reload. `docker compose exec redis redis-cli KEYS "*dirty*"` returned empty array. Effect: daemon continued using 900s timeout, claude CLI kept timing out at 15 minutes despite the new config value. Aleksei had to restart the container manually.

This is a discrepancy in user mental model. "I edited the config file, so config is updated" is a natural expectation. The current implementation breaks that expectation.

The fix is to add a file watcher (or polling-based checksum check) on `config.yml` so any change to the file triggers the same reload mechanism that the UI uses.

## Scope

Add file change detection for `config.yml`. On detected change, set the existing dirty_key in Redis. The existing reload mechanism (`_reload_config_from_dirty_key` or equivalent in runner.py:316 area) handles the rest.

Choose between two implementation approaches: file watcher (inotify-based via watchdog library) OR polling with checksum (lighter, no new deps, simpler). Recommend polling for simplicity unless team prefers watcher.

This PR does NOT change the existing reload mechanism semantics. It does NOT add new config fields or change config schema. It only ADDS a new trigger source for the existing reload path.

## Files to touch

`src/daemon/main.py` (or wherever the daemon main loop / startup lives): on startup, record the initial mtime + size + sha256 of `config.yml`. Periodically (every 5 seconds, configurable) check if any of these have changed. If yes, write to Redis the same `dirty_key` that the UI Settings endpoint writes (find this key by searching for where it is set in src/web/).

Sketch:

```python
import asyncio
import hashlib
import os
from pathlib import Path

CONFIG_PATH = Path(os.environ.get("PO_CONFIG_PATH", "config.yml"))
CONFIG_WATCH_INTERVAL_SEC = 5


async def watch_config_file_changes(redis_client):
    """Poll config.yml for changes; on change, set dirty_key flag."""
    if not CONFIG_PATH.exists():
        return  # nothing to watch (e.g., test env without file)
    last_signature = _config_signature(CONFIG_PATH)
    while True:
        await asyncio.sleep(CONFIG_WATCH_INTERVAL_SEC)
        if not CONFIG_PATH.exists():
            continue
        current_signature = _config_signature(CONFIG_PATH)
        if current_signature != last_signature:
            last_signature = current_signature
            await _set_config_dirty_flag(redis_client)


def _config_signature(path: Path) -> tuple[float, int, str]:
    """Cheap-then-strong: mtime + size, escalate to sha256 if both change."""
    stat = path.stat()
    return (stat.st_mtime, stat.st_size, _compute_sha256(path))
```

The signature uses sha256 to avoid false positives from `touch` or atomic-write tools that change mtime without actual content change. SHA computation on a small config file is sub-millisecond — overhead is negligible.

`_set_config_dirty_flag` writes the same Redis key that the UI uses. Implementer should grep for where the UI writes this key (likely in src/web/app.py or src/web/settings_handler.py) and reuse the exact same key name and format.

Hook the watcher coroutine into the main daemon startup so it runs as a background task alongside the runner loop. Use `asyncio.create_task` and ensure clean cancellation on shutdown.

`src/web/`: NO changes. UI continues to write the same dirty_key as today. The new file watcher is parallel.

`tests/test_config_file_watcher.py` (new): unit test that:
- Initializes watcher with a temp config file
- Verifies initial signature is computed
- Modifies the file
- Verifies dirty_key is set in mocked Redis after one watch interval
- Verifies that touching the file without content change does NOT trigger (sha256 unchanged)
- Verifies that file deletion does not crash the watcher

`tests/test_runner.py` or wherever the runner startup is tested: add assertion that the watcher coroutine is spawned alongside the main loop.
