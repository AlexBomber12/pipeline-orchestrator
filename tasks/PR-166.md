# PR-166: Coder ESCALATE protocol in FIX cycle

Branch: pr-166-coder-escalate-protocol
- Type: feature
- Complexity: medium
- Depends on: PR-164
- Priority: 1
- Coder: any

## Problem

When a coder is in a FIX cycle and concludes that the failure is genuinely outside the coder's mandate to fix — for example, infrastructure errors (rate limit, DNS, secrets), test failures requiring changes to production code beyond stated PR scope, contradictions between task spec and existing architecture, or simply confusion after multiple unsuccessful attempts — the coder currently has no way to communicate this back to the daemon explicitly.

The existing recovery mechanisms (PR-164 no-push counter, fix_iteration_cap counter) catch the SYMPTOMS of stuck coders (no productive push, too many iterations), but they require N attempts before triggering. A coder who recognizes "I cannot fix this" on the first cycle has no faster exit — must waste 3 retry cycles to be parked.

This is also a SAFETY issue. If CI is failing because of an infrastructure problem (e.g., GitHub rate limit exhausted from PR-163, DNS misconfiguration on test runner, expired secrets), a coder might attempt to "fix" it by modifying production code that has nothing to do with the actual failure. This is the worst-case outcome — a misdirected "fix" that adds bugs while the real issue is environmental.

The fix is an explicit ESCALATE protocol: the coder can output a structured marker on stdout indicating "I cannot fix this; here is why; please escalate." Daemon parses the marker, posts an explanatory comment on the PR, applies the existing `escalated` label, transitions runner to IDLE. The PR is parked for human inspection.

This complements PR-164 (mechanical no-push circuit breaker) with a SEMANTIC circuit breaker (coder-initiated). Both feed into the same downstream escalation machinery.

## Scope

Three combined changes:

1. **AGENTS.md update**: document the ESCALATE protocol — when to use it, what to output, what happens after.
2. **Daemon parsing in fix.py**: detect `ESCALATE: <reason>` marker in coder stdout and route to escalation path.
3. **Reuse existing escalation machinery**: leverage the `_escalate_fix_iteration_cap` infrastructure (label, comment, IDLE transition, `is_escalated` flag) without code duplication.

This PR does NOT modify the existing fix_iteration_cap or no_push_fix_count counters. It does NOT change the IDLE transition semantics. It adds a NEW trigger source for the same downstream escalation path.

Content sketch:

```markdown
## ESCALATE protocol (when coder cannot fix)

If during a FIX cycle you determine that the failure is genuinely outside your
mandate to fix, output an ESCALATE marker on the LAST line of stdout. The
marker format is exactly:

  ESCALATE: <one-line reason>

Examples of reasons that warrant ESCALATE (MUST):

- CI logs show "rate limit exceeded" for GitHub API
- CI logs show DNS resolution failure or network unreachable
- CI logs show "secret not found" or auth errors with GitHub
- Test failure requires changes to production code beyond the stated PR scope
  (Files NOT to touch list)
- Task spec contradicts existing code architecture (ambiguity, not a bug)
- Three attempts at the same fix failed with the same error — likely root
  cause is not what you assumed

Examples of reasons that do NOT warrant ESCALATE (just fix):

- Banal syntax errors or typos
- Test assertion failures with clear root cause
- Linting or formatting issues
- Missing imports or simple type errors

Examples of reasons that MAY warrant ESCALATE (use judgment):

- Test passes locally but fails in CI consistently → infra suspect
- Coverage check fails for code already tested elsewhere → tooling issue
- Behavior matches spec but spec was wrong → architectural decision needed

After ESCALATE, the daemon will:
1. Post a comment on the PR with your reason text
2. Apply the `escalated` label to the PR
3. Transition runner to IDLE and stop further FIX cycles on this PR
4. Wait for human review before any further action

To resume work after human resolves the issue, click Resume in the dashboard
or close + reopen the PR. The escalated label and is_escalated flag are
cleared at that point.
```

`src/daemon/handlers/fix.py`: in `handle_fix`, after coder subprocess completes, parse stdout for ESCALATE marker BEFORE the existing push detection logic. Sketch:

```python
def _parse_escalate_marker(stdout: str) -> str | None:
    """Return the ESCALATE reason if stdout's last non-empty line starts
    with 'ESCALATE: '; else None."""
    for line in reversed(stdout.splitlines()):
        line = line.strip()
        if not line:
            continue
        if line.startswith("ESCALATE:"):
            return line[len("ESCALATE:"):].strip()
        return None  # last non-empty line was not ESCALATE
    return None
```

In handle_fix flow, after coder exits and stdout is captured:

```python
escalate_reason = _parse_escalate_marker(stdout)
if escalate_reason:
    await self._escalate_fix_coder_initiated(current_pr, escalate_reason)
    return
# ... existing push detection logic continues
```

`tests/test_fix_handler.py`: add tests covering:
- Coder stdout ends with `ESCALATE: rate limit exceeded` → escalation path triggered, label applied, comment posted, state IDLE.
- Coder stdout has `ESCALATE: ...` in middle, last line is something else → NOT triggered (only last non-empty line counts).
