# PR-177: HTMX whitelist 400 status for validation error display

## Problem

Backend correctly returns HTTP 400 for upload validation errors (`src/web/app.py:_render_upload_error` with `status_code=400` and `HX-Retarget` header for proper HTMX swap target). However the HTMX client in `src/web/templates/base.html` only whitelists 404, 422, 503 in its `htmx:beforeSwap` handler:

```js
document.addEventListener('htmx:beforeSwap', function (evt) {
    const status = evt.detail.xhr.status;
    if (status === 404 || status === 422 || status === 503) {
        evt.detail.shouldSwap = true;
        evt.detail.isError = false;
    }
});
```

When the user uploads a zip with invalid task headers (Type missing, invalid enum value, malformed Branch line, etc.), validator returns 400 with rendered error HTML. HTMX silently discards the response because 400 is not whitelisted. User sees nothing — upload appears to "do nothing." User has no idea why.

This is a recurring source of user confusion. Validator errors are detailed and helpful, but invisible.

## Scope

Add 400 to the HTMX status whitelist in `base.html`. Add unit test asserting the whitelist behavior.

This PR does NOT change validator logic. It does NOT change error message format. It only fixes the gap where backend speaks but frontend doesn't listen.
