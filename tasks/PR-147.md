# PR-147: Re-trigger @codex review on stale CHANGES_REQUESTED

Branch: pr-147-stale-review-retrigger
- Type: bugfix
- Complexity: medium
- Depends on: none
- Priority: 2
- Coder: any

## Problem

When a PR has review status CHANGES_REQUESTED and the coder pushes a fix, the expected flow is: new push invalidates the old review, Codex reviews the new head automatically, daemon observes the new review and acts. In practice the daemon sometimes ends up in FIX or WATCH with:

- `state.current_pr.review_status == ReviewStatus.CHANGES_REQUESTED`
- Head SHA has advanced since the review (new pushes after the CHANGES_REQUESTED state)
- No automatic fresh review comment from Codex

Daemon waits indefinitely for the fresh review that never arrives, because nobody posted `@codex review` to trigger one on the new head. Operator has to manually post `@codex review`. Documented in the pre-Phase 1 sprint observations.

The `_post_codex_review` dedup in `src/daemon/handlers/hung.py` is head-aware via `has_recent_codex_review_request(after_iso=head_commit_date)` in `src/github_client.py`. It is safe to call from a stale-detection path: if a review was already requested on the current head, dedup blocks the repost; if not, the trigger goes through.

## Scope

Add a stale-review detector in the WATCH handler (the state that polls for reviews). On each poll iteration, check:

1. `state.current_pr is not None`
2. `state.current_pr.review_status == ReviewStatus.CHANGES_REQUESTED`
3. Fetch head SHA and push time via github_client:
   - `pr_meta = github_client.get_pr_metadata(owner_repo, pr_number)` returns `{author, head_sha, head_commit_date}`
   - `last_push_time = github_client.get_branch_last_push_time(...)` (existing helper at src/github_client.py:610)
4. `now - last_push_time >= stale_review_threshold_min` (push has settled, not in-flight)
5. Debounce: `state.last_stale_retrigger_at is None` OR `now - last_stale_retrigger_at >= 1 hour`

On all-true: call `self._post_codex_review(pr_number)` (inherited from HungMixin). The mixin's internal dedup handles the case where a review was already requested on the current head. Record the attempt via `state.last_stale_retrigger_at = now` regardless of dedup outcome. Log event verbatim: `Stale CHANGES_REQUESTED on PR #{n}; re-triggering @codex review.`

### Config

- `stale_review_threshold_min: int = 10` in the daemon config
- Env var override `PO_STALE_REVIEW_THRESHOLD_MIN`

Threshold of 10 minutes is a starting guess. If retrigger fires too aggressively on normal fix cycles, raise to 15 or 20 in a follow-up.

### Debounce

Hard rate limit: 1 retrigger per hour per PR even if conditions stay true. Prevents spam if something else is broken. Field `last_stale_retrigger_at: datetime | None` on `RepoState` (not on PRInfo, so it survives PR transitions; reset to None on new PR).
