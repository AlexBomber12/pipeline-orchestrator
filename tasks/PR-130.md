# PR-130: Deduplicate consecutive identical events in the event log

Branch: pr-130-event-log-dedup
- Type: feature
- Complexity: low
- Depends on: none
- Priority: 2
- Coder: any

## Problem

The event log fills with identical messages during long IDLE or WATCH cycles. Observed pattern: 9+ consecutive `IDLE No tasks available` entries, one per poll tick (~60 seconds apart). Current history retains 100 events; in a 10-minute idle window most of that buffer is duplicates, crowding out real transitions.

The fix is standard log deduplication: consecutive events with the same state + message get folded into one entry with a counter and last-seen timestamp.

## Scope

Add dedup logic at event append time. When `log_event()` is called:
1. Look at the most recent event in history.
2. If state and message match exactly, increment a `count` field and update `last_seen_at` on that event. Do not append a new entry.
3. If different, append a new event normally with `count=1` and `last_seen_at == created_at`.

Render in the UI as:
- If `count == 1`: render as before (`{ago} {state} {message}`).
- If `count > 1`: render as `{first_seen_ago} -> {last_seen_ago}  {state}  {message} (x{count})`.

Do not change the maximum history size (still 100 events). Dedup is about not wasting slots on duplicates, not about capping storage.

The dedup applies ONLY to consecutive identical events. A-A-B-A is three distinct entries (A with count=2, then B, then A with count=1).

## Files to modify

- `src/models.py`: add `count: int = 1` and `last_seen_at: datetime` fields to the Event model (or whatever type holds log entries).
- `src/daemon/runner.py` or wherever `log_event` lives: implement the dedup check before appending.
- `src/web/templates/components/event_log.html` (and `event_list.html` if separate): update rendering to show count and last-seen when count > 1.
- Tests: dedup-increment test, no-dedup-different-state test, count-resets-after-different-event test, UI render test for count > 1.

## Success criteria

- 9 consecutive `IDLE No tasks available` events get folded into one entry showing `(x9)` and a last-seen timestamp.
- A different intervening event resets the dedup; next identical event starts fresh.
- History size still capped at 100 distinct entries.
- UI renders count and last-seen clearly without cluttering the single-event case.
- `scripts/ci.sh` exits 0.
- `ruff check src tests` exits 0.
- Coverage stays at 100%.
