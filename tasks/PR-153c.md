## Problem

PR-153a established the Python skeleton, PR-153b added the docker compose test stack. This PR provides the remaining pieces to make the smoke test runnable end-to-end: the test config file (config.test.yml with fast polling intervals and the testbed repo), a separate test requirements file (requirements-test.txt with pytest-playwright and requests), and a small change to src/config.py to honor a PO_CONFIG_PATH environment variable for selecting which config file the daemon and web load.

After this PR merges, `pytest tests/e2e/` against the running test stack (started via `docker compose -f docker-compose.test.yml up -d` from PR-153b) runs the smoke test from PR-153a successfully and exits 0. This closes Sprint F1.0 part 1 (infrastructure) and unblocks PR-154 series (behavioral tests with mock coder shim).

## Scope

Add config.test.yml at repo root with test-tuned values. Add requirements-test.txt declaring pytest-playwright, playwright, and requests. Modify src/config.py to read PO_CONFIG_PATH from environment with fallback to the existing default. Update the smoke test in tests/e2e/test_smoke.py from PR-153a to actually exercise the dashboard (was placeholder 1+1==2; becomes a real HTTP check against /api/states).

The config override is the smallest possible change: a one-line addition in src/config.py wherever the config path constant is defined. Existing call sites that pass a hardcoded path remain valid (the env var override does not affect them); call sites that pass no path get the env var override behavior. The change is non-breaking and unit-test backwards-compatible.

## Files to create

`config.test.yml`: at repo root. YAML file with the same schema as config.yml but tuned for tests.

```
repositories:
  - url: https://github.com/AlexBomber12/pipeline-orchestrator-testbed.git
    branch: main
    auto_merge: true
    poll_interval_sec: 2

daemon:
  poll_interval_sec: 2
  review_timeout_min: 2
  auto_fallback: true
  hung_fallback_codex_review: true
  error_handler_use_ai: false
  claude_model: opus
  fix_idle_timeout_sec: 60
  planned_pr_timeout_sec: 60
  coder_priority:
    codex: 81
    claude: 76
  exploration_epsilon: 0.15
  rate_limit_session_pause_percent: 90
  rate_limit_weekly_pause_percent: 95

web:
  host: 0.0.0.0
  port: 8000

auth:
  claude_config_dir: /data/auth/claude
  gh_config_dir: /data/auth/gh
```

Notes for the implementer:
- error_handler_use_ai is false in the test config so that diagnose_error CLI is not invoked during tests (saves quota and avoids OBS-4 patterns).
- review_timeout_min is 2 minutes (vs 60 in production) so Codex review timeout is observable in test wall-clock.
- planned_pr_timeout_sec and fix_idle_timeout_sec are 60 seconds (vs 1800 in production) so daemon reaches HUNG quickly in tests that probe timeout behavior.
- rate_limit_session_pause_percent and rate_limit_weekly_pause_percent use the new field names from the config split, NOT the deprecated rate_limit_pause_percent. The test config does not need to maintain backward compatibility with old field names.
- If the schema validator (Pydantic models in src/config.py) rejects unknown fields and the test config has a typo, the daemon will fail to start. Run docker compose -f docker-compose.test.yml up -d after creating this file to verify.

`requirements-test.txt`: at repo root.

```
pytest-playwright==0.5.2
playwright==1.49.1
requests==2.32.3
```

Pinning specific versions so the test environment is reproducible across host machines and across PR-155a's GitHub Actions runners. Do not depend on requirements.txt entries here; the file should be self-contained for `pip install -r requirements-test.txt` to install everything needed.

## Files to touch

`src/config.py`: locate the function or module-level constant that determines the config file path. Likely candidates: a CONFIG_PATH constant, a load_config function, or a path argument in the Config class. Modify so that the path resolution honors the PO_CONFIG_PATH environment variable.

Concrete change pattern (adapt to actual code structure):

If there is a function `load_config(path: str = "config.yml") -> AppConfig`, change to:
```
def load_config(path: str | None = None) -> AppConfig:
    resolved_path = path or os.environ.get("PO_CONFIG_PATH") or "config.yml"
    # rest of body unchanged, using resolved_path
```

If there is a module-level CONFIG_PATH constant used by load_config(), inject env var resolution at the constant site:
```
CONFIG_PATH = os.environ.get("PO_CONFIG_PATH", "config.yml")
```

The implementer should choose whichever pattern is least invasive. The semantic is the same: when PO_CONFIG_PATH is set in the environment (which happens for daemon-test and web-test in docker-compose.test.yml), the daemon reads config.test.yml; otherwise it falls back to config.yml as before.

Add necessary import (`import os` if not already imported).

`tests/e2e/test_smoke.py` (created in PR-153a, was placeholder): replace the body with a real smoke test that uses fixtures. The test function `test_smoke_dashboard_responds` takes the dashboard_url and testbed_slug fixtures. It performs:
- urllib.request.urlopen(f"{dashboard_url}/api/states", timeout=5), assert status 200
- Parse the response JSON, assert it is a list (not an error envelope)
- Assert at least one entry has slug equal to testbed_slug (the testbed repo from config.test.yml)
- Assert the entry has a state value among the known set: IDLE, CODING, WATCH, FIX, MERGE, ERROR, HUNG, PAUSED, PREFLIGHT

The test does NOT exercise auth-status, upload, or anything that requires gh/claude/codex CLI to be authenticated in the test container. Auth status is checked indirectly: if /api/states returns 200 with a valid entry, the dashboard is up and the daemon registered the testbed repo. That is enough for the smoke.

`tests/e2e/README.md` (created in PR-153a): update the Status section to indicate Sprint F1.0 part 1 is complete and behavioral tests come in PR-154 series. Two-line update.

## Success criteria

1. `pip install -r requirements-test.txt` exits 0 in a fresh venv. The three packages install without conflicts against an environment that already has requirements.txt installed (no version conflicts on shared transitive deps like httpx).

2. With the test stack running (PR-153b's `docker compose -f docker-compose.test.yml up -d`), running `docker compose -f docker-compose.test.yml logs daemon-test --tail=20` shows the daemon successfully loaded config.test.yml: log lines indicate "Daemon starting with 1 repositories" and the URL is the testbed repo.

3. With the test stack running, `curl -sS http://localhost:18800/api/states | jq` returns a JSON array containing one entry with slug "AlexBomber12__pipeline-orchestrator-testbed".

4. With the test stack running and dependencies installed, `pytest tests/e2e/ -v` collects exactly one test (test_smoke_dashboard_responds) and that test PASSES.

5. With the prod stack running and the test stack stopped, `pytest tests/e2e/ -v` fails the smoke test cleanly (urllib raises ConnectionError on port 18800 because nothing is listening there). The failure is reported as a test failure, not a pytest collection error.

6. `python -m pytest -q` (default invocation, scripts/ci.sh second step) still collects ONLY tests under tests/ excluding tests/e2e/. The norecursedirs setting from PR-153a is unchanged.

7. With PO_CONFIG_PATH unset in the environment, the production daemon (docker-compose.yml) starts and loads config.yml as before. This is verified by stopping and restarting the prod stack and checking the daemon's "Daemon starting with N repositories" log line still references the production repos from config.yml.

8. With PO_CONFIG_PATH=/app/config.test.yml set in the daemon-test container's environment, the test daemon loads config.test.yml. Verified in success criterion 2.

9. Existing src/config.py unit tests (whatever they assert) continue to pass at 100% coverage. Add a unit test in the same test module that asserts: when PO_CONFIG_PATH is set in os.environ, load_config without arguments returns a config loaded from that path. Use monkeypatch fixture to set and unset the env var.

10. scripts/ci.sh exits 0. ruff check src tests passes. Coverage stays at 100% (the new unit test for env var override covers the new branch).

11. config.test.yml uses rate_limit_session_pause_percent and rate_limit_weekly_pause_percent (the new split field names), not the deprecated single rate_limit_pause_percent. If src/config.py raises a deprecation warning on the new fields, that is a bug in src/config.py to fix in a follow-up; this PR uses the new fields as authoritative.
