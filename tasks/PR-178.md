# PR-178: Pulse animation jump fix (smooth opacity transition)

- Complexity: low
- Depends on: none

- Coder: any

## Problem

The `pulse-dot` animation in `src/web/templates/base.html` is implemented as a CSS keyframe animation:

```css
@keyframes pulse-dot {
    0%, 100% { opacity: 1; }
    50% { opacity: 0.3; }
}

.pulse-dot {
    animation: pulse-dot 2s ease-in-out infinite;
}
```

When a state badge that has `pulse-dot` class is dynamically swapped via HTMX (state transition triggers re-render of `repo_cards.html` or `repo_summary.html`), the animation **restarts from frame 0**. Visually this produces a jarring "jump" — the dot is mid-fade at opacity 0.7 in old DOM, then suddenly opacity 1.0 in new DOM. User perceives this as a flicker or jolt.

The fix is to use CSS transition instead of keyframe animation. Transitions interpolate between current value and target value, so a re-rendered element starts from wherever it visually was.

## Scope

Replace the keyframe-based pulse animation with a CSS-transition-based approach. Test that re-rendering a pulsing badge does not produce visual jump.

This PR does NOT change which elements get `pulse-dot` class. It does NOT change the visual rhythm (still ~2 seconds period). It only changes the underlying CSS technique.

## Files to touch

`src/web/templates/base.html`: replace the keyframe animation with transition-based approach. There are several common patterns:

**Approach A: Two-state CSS animation with `animation-fill-mode: both`**

```css
@keyframes pulse-dot {
    0% { opacity: 1; }
    50% { opacity: 0.3; }
    100% { opacity: 1; }
}

.pulse-dot {
    animation: pulse-dot 2s ease-in-out infinite;
    animation-fill-mode: both;
    /* This makes the element "remember" its visual state across DOM swaps */
}
```

This may not fully fix the issue because keyframes still restart on re-render.

**Approach B: JavaScript-driven transition (more reliable for HTMX swaps)**

```css
.pulse-dot {
    opacity: 1;
    transition: opacity 1s ease-in-out;
}

.pulse-dot.dim {
    opacity: 0.3;
}
```

```js
setInterval(() => {
    document.querySelectorAll('.pulse-dot').forEach(dot => dot.classList.toggle('dim'));
}, 1000);
```

This drives the pulse via global timer — every dot pulses in sync, but state survives HTMX swap because the transition picks up from current opacity.

**Approach C: SVG animation with `<animate>` element**

More complex; probably overkill for a 10px dot. Mention but recommend NOT using.

**Recommended: Approach B (JS-driven transition).** Implement this. Keep code simple — single `setInterval` toggling `.dim` class on all `.pulse-dot` elements every 1000ms. The `transition: opacity 1s ease-in-out` smoothly interpolates.

When element re-rendered via HTMX swap, the new element starts at default opacity 1 and the next interval tick will toggle `.dim` (or not, depending on parity). No visual jump because there's no abrupt opacity change — just transition from 1.0 to 0.3 over 1 second smoothly.

`tests/test_web_pulse_animation.py` (new, optional): unit test asserting the CSS contains `transition` rule on `.pulse-dot` and the global timer is set up. Can be a simple regex check on base.html source.
