# PR-048: Fail-closed for GitHub observation errors

Branch: pr-048-fail-closed-observation

- Type: bugfix
- Complexity: low

## Problem

`_has_new_codex_feedback_since_last_push` in runner.py returns True if the GitHub API call fails. Observation failure is interpreted as "yes, new feedback exists", which triggers handle_fix - a mutating action that runs Claude CLI and pushes commits.

Failures in observation should not automatically produce repository changes. A transient GitHub outage could trigger unnecessary FIX cycles that consume Claude tokens and may push incorrect fixes.

## Root cause

`src/daemon/runner.py:2194-2195`:
```python
except Exception:
    return True
```

The function returns a binary True/False but the third outcome - "we don't know because the API failed" - is silently collapsed into True.

## Scope

Change the function to return a 3-state result. On API failure, callers stay in WATCH and retry on the next cycle instead of entering FIX.

## Changes

### src/daemon/runner.py

Replace the return type of `_has_new_codex_feedback_since_last_push` with an enum:

```python
class FeedbackCheckResult(Enum):
    NEW = "new"           # Codex posted after last push
    NONE = "none"         # No Codex activity after last push
    UNKNOWN = "unknown"   # API call failed, cannot determine
```

Update the function:
- On successful check with new feedback -> NEW
- On successful check with no new feedback -> NONE
- On exception -> UNKNOWN (log the exception, do not raise)

Update the caller in `handle_watch` (around line 1476):
```python
elif review == ReviewStatus.CHANGES_REQUESTED:
    result = self._has_new_codex_feedback_since_last_push()
    if result == FeedbackCheckResult.NEW:
        await self.handle_fix()
        return
    if result == FeedbackCheckResult.UNKNOWN:
        self.log_event(
            f"PR #{found.number} CHANGES_REQUESTED but feedback check "
            "failed; staying in WATCH, will retry next cycle"
        )
        return
    # NONE: no new feedback since last push, fall through to timeout check
    self.log_event(
        f"PR #{found.number} CHANGES_REQUESTED but no new "
        "Codex feedback since last push; waiting for fresh review"
    )
```

When UNKNOWN, return early (don't fall through to HUNG timeout check either). Transient API errors shouldn't accelerate a PR toward HUNG.

### tests/test_runner.py

Add tests:
- test_feedback_check_returns_unknown_on_api_failure: mock GitHub API to raise, expect FeedbackCheckResult.UNKNOWN
- test_handle_watch_stays_in_watch_on_unknown_feedback: CHANGES_REQUESTED + UNKNOWN from feedback check, expect state remains WATCH, no FIX called
- test_handle_watch_skips_hung_timeout_on_unknown: CHANGES_REQUESTED + UNKNOWN + elapsed > timeout_min, expect state stays WATCH not HUNG (because observation is unreliable, not because the PR is actually hung)

Update existing tests that mock `_has_new_codex_feedback_since_last_push` to return the new enum values instead of True/False.

## Success criteria
- scripts/ci.sh exits 0
- New tests pass
- All existing tests pass (with updated return value assertions)
- On GitHub API failure during feedback check, runner stays in WATCH
- Runner does not flip to HUNG solely due to observation failures
- Log event clearly distinguishes UNKNOWN from NONE
