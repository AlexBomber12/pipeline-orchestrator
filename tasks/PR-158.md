# PR-158: Coder pin from task header respected (FINDING-2)

Branch: pr-158-coder-pin-respected
- Type: bugfix
- Complexity: medium
- Depends on: none
- Priority: 1
- Coder: any

## Problem

Task files have a `Coder:` header field accepting `claude` / `codex` / `any`. The validator in `src/queue_parser.py` parses this field correctly into `TaskHeader.coder` (line 56 dataclass field, line 259 parsing, line 277 instantiation). However the selector at `src/daemon/selector.py:31-52` ignores `header.coder` entirely. It consults only `ctx.repo_config.coder` (the per-repo dropdown in Settings) and `ctx.app_config.daemon.coder` (the global default).

This produces two bugs:

1. **Happy-path bug.** Even when both Claude and Codex are healthy, a task with `Coder: codex` may run on Claude if the repo's selector chooses Claude. User intent in the task file is silently dropped.
2. **FINDING-2 (unavailable coder fallback).** When the pinned coder is unavailable (rate-limited, broken, missing auth), the task falls through to another coder via `auto_fallback`. User receives a Claude-implemented PR for a task explicitly pinned to Codex.

The fix is to extend the selector so it honors `header.coder` when it is `claude` or `codex` (NOT when it is the default `any`, which means "use whatever the repo/global default chooses").

When the task header pin is hard (`claude` or `codex`) and that coder is unavailable, the task transitions to HUNG with a clear message rather than falling back. This matches the FINDING-2 fix specified in `docs/roadmap.md` lines 62-67.

## Scope

Modify `src/daemon/selector.py` to consume task header coder pin. Modify the IDLE handler to pass the active task header into the SelectionContext. Add unit tests for selector with task pin combinations. Update one or two call sites that construct SelectionContext.

This PR does NOT change `repo_config.coder` semantics (per-repo Settings dropdown). That field still acts as the default when no task header pin is set. The precedence becomes: task header pin > repo_config pin > app_config default.

This PR does NOT change auto_fallback behavior for unpinned tasks (`Coder: any`). Those tasks continue to fall back as before.

## Files to touch

`src/daemon/selector.py`: extend `SelectionContext` dataclass with a new optional field `task_coder_pin: str | None = None`. Default None. Modify `eligible_coders(ctx)` so the precedence is:

1. If `ctx.task_coder_pin` is set to `claude` or `codex` (NOT `any`, NOT None), that is the hard pin.
2. Else if `ctx.repo_config.coder` is set, that is the soft pin (current behavior).
3. Else `ctx.app_config.daemon.coder` is the default.

When task pin is hard:
- Return ONLY that coder if eligible (passes `_supports_runtime`, not rate-limited, auth ok, not disabled for repo).
- If hard-pinned coder is NOT eligible, return empty list. Caller (selector) returns None. Caller of selector (IDLE handler) detects None and transitions to HUNG with message.

The current implementation at line 33-52 mixes preference with auto-fallback. The change is to short-circuit the loop when task pin is hard.

Sketch (illustrative, not literal):

```python
def eligible_coders(ctx: SelectionContext) -> list[str]:
    task_pin = ctx.task_coder_pin
    if task_pin in ("claude", "codex"):
        if _coder_runtime_ok(task_pin, ctx):
            return [task_pin]
        return []  # hard pin unavailable -> caller transitions to HUNG
    # ... existing logic ...
```

Where `_coder_runtime_ok(name, ctx)` is a small helper that runs the same checks (`_supports_runtime`, `_is_rate_limited`, `_auth_failed`, `_is_disabled_for_repo`) on a single coder. Extract from the existing loop to avoid duplication.

`src/daemon/handlers/idle.py`: where the IDLE handler builds `SelectionContext` to call `select_coder`, pass `task_coder_pin=current_task_header.coder if current_task_header else None`. Find the construction site by searching for `SelectionContext(` in the file. The exact call site is wherever the daemon evaluates "which coder spawns this task."

When `select_coder` returns None and the active task has a hard pin, the IDLE handler must transition the repo state to HUNG with `error_message = f"Task {pr_id} pinned to {pin} but coder unavailable"`. Do NOT fall back to another coder. Do NOT silently skip the task.

`tests/test_selector.py`: add unit tests covering:
- Task pin claude + Claude eligible → returns [claude]
- Task pin claude + Claude rate-limited → returns []
- Task pin codex + Codex auth failed → returns []
- Task pin any + repo pin claude → returns [claude] then others (existing behavior)
- Task pin None + repo pin codex → returns [codex] then others (existing behavior)

`tests/test_idle_handler.py` (or wherever IDLE handler tests live): add test that when `select_coder` returns None due to hard task pin unavailable, repo transitions to HUNG with the expected error message.
