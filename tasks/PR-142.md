# PR-142: Clear upload feedback lifecycle

Branch: pr-142-upload-feedback-lifecycle
- Type: feature
- Complexity: medium
- Depends on: none
- Priority: 2
- Coder: any

## Problem

The current upload flow has minimal visual feedback. When the user clicks Upload tasks and selects a file, several things can happen silently:
1. Upload takes 1-3 seconds (zip upload + validation), no loading indicator shown.
2. On success the user sees a generic "Tasks queued. The daemon will commit and push on its next cycle." - does not say how many files accepted, which ones, or when they will actually appear in the repo.
3. On error (e.g. the recent 400 from invalid task type `chore`), the error banner says only the outer error, not the specific file and field that failed validation.
4. Success message never auto-dismisses. After a minute the user does not know if the daemon has picked up the work or if something stalled.

The div that receives feedback is named `upload-error-{name}` regardless of success - a naming leak that causes minor confusion in the HTML source.

This PR implements a clear four-stage upload feedback lifecycle.

## Scope

### Stage 1: Loading (between click and response)

Show an HTMX-indicator spinner next to the Upload tasks label. Use `hx-indicator="#upload-indicator-{name}"`. The indicator is a small inline spinner (SVG or CSS animation) with text "Uploading..." that is hidden by default and visible only during the request.

Disable the file input during upload to prevent double-submit.

### Stage 2: Success (HTTP 200)

Rename the target div from `upload-error-{name}` to `upload-feedback-{name}`. Update both the form's `hx-target`, the div's `id`, and all places in `src/web/app.py` that set `HX-Retarget` headers.

The success message should be detailed:
```
Accepted 18 task files (PR-124 through PR-141).
Daemon will commit and push within 60 seconds.
Auto-dismissing in 30 seconds.
```

Include a dismiss button (×) in the top-right of the success card so users can close manually. Auto-dismiss via HTMX `hx-on::after-settle` + `setTimeout` that removes the element after 30 seconds.

If the upload includes a QUEUE.md or non-task helper files (AGENTS.md, CLAUDE.md), mention them separately in the message.

### Stage 3: Error (HTTP 400/422/500)

Error messages must include:
- The outer failure reason
- If validation failed on a specific file: the filename and the field that was invalid (e.g. "PR-124.md: invalid Type 'chore'; expected one of ['architecture', 'bugfix', 'config', 'docs', 'feature', 'refactor', 'ux']")
- If the zip contained invalid paths or too many files: the specific rejection reason

Update `src/web/app.py` to wire the specific validation error message from `parse_task_header` into the rendered error message, not just the HTTP status.

Error card does not auto-dismiss. User must click × to close, or it disappears when the next upload attempt replaces the content.

### Stage 4: Commit confirmation (via event log)

After the daemon drains the Redis manifest and commits the files on its next IDLE cycle, it should log a specific event: `Uploaded N task files to tasks/ and pushed to main`. This event is already logged partially (there is a "Pushed uploaded task files: [...]" entry in the current codebase) - verify it is present, add if missing, and ensure it surfaces in the visible event log on the repo detail page.

This is the final confirmation that the files are actually in the repo. The user sees the lifecycle: Upload -> Accepted -> Committed -> Task picked up.

## Files to modify

- `src/web/templates/components/repo_cards.html`: rename `upload-error-{name}` to `upload-feedback-{name}`, add `hx-indicator` and the spinner element, add loading text.
- `src/web/templates/components/upload_success.html`: add dismiss button, auto-dismiss timer, detailed message formatting.
- `src/web/templates/components/upload_error.html`: add dismiss button, ensure detailed validation errors render.
- `src/web/app.py`: rename all `HX-Retarget: #upload-error-...` to `#upload-feedback-...`. Enrich the success message to include file count, PR ID range, and estimated daemon cycle time. Ensure specific validation errors from `parse_task_header` propagate into the error message (not lost in a generic 400 response).
- `src/daemon/handlers/idle.py` (or wherever `_drain_upload_manifest` lives): verify the post-commit event log line is present and informative.
- Tests: cover the indicator attribute in the form, success message format with multiple files and with single-file zip, error message format for each known validation failure category, dismiss button click handler.

## Success criteria

- Clicking Upload tasks on a slow connection shows "Uploading..." spinner until response arrives.
- Success response shows file count, PR ID range, and estimated time to daemon pickup.
- Success message auto-dismisses after 30 seconds or on manual × click.
- Error response includes the specific filename and field that failed (e.g. invalid Type value with the list of allowed values).
- Error message does not auto-dismiss.
- After daemon drains the manifest, the event log on the detail page shows a clear line about the commit and push.
- All HTML id attributes for the feedback div use the new name `upload-feedback-{name}`.
- No regression: existing upload paths (happy path, zip with valid task files, single .md file) still work.
- `scripts/ci.sh` exits 0.
- `ruff check src tests` exits 0.
- Coverage stays at 100%.
