# PR-155a: GitHub Actions CI workflow with two-tier gate

## Problem

Sprint F1.0 part 1 (PR-153 series) and part 2 (PR-154 series) deliver the test infrastructure and behavioral tests that protect the four production behaviors validated during testing-week. They run locally only. Without an automated runner, real merge gating depends on a human remembering to run the suite. The four protected behaviors plus everything Sprint F1.1+ adds are protected only as long as that discipline holds.

The chosen solution is GitHub Actions running on every pull_request and push to main. The workflow has two jobs. The unit job runs scripts/ci.sh in a clean environment. The integration job runs the e2e suite from PR-153/154 with the mock coder shim. Both must be green for merge eligibility, alongside the existing Codex review +1 contract.

The integration job needs GitHub authentication scoped to the testbed repository for real git push, PR open, and comment operations. The chosen credential mechanism is a GitHub App with installation on the testbed repo, generating ephemeral 1-hour tokens per workflow run via actions/create-github-app-token. This avoids long-lived PATs that require manual rotation and provides finer-grained permissions than a user PAT.

This PR delivers ONLY the workflow YAML file. The setup documentation for the GitHub App is in PR-155b. The AGENTS.md update is in PR-155b. Scripts/test-e2e.sh and docs/local-e2e.md are in PR-155c.

## Scope

Add `.github/workflows/ci.yml`. No other file changes.

The PR is permitted to fail its own integration job on first run because the GitHub App secrets (TESTBED_APP_ID, TESTBED_APP_PRIVATE_KEY) are not yet configured at the moment of merge. Branch protection enforcement of the new checks is enabled by the human operator AFTER merge as part of the procedure documented in PR-155b. Merge of this PR therefore requires manual override of the integration check; the PR description must explicitly note this and instruct the human reviewer.

## Files to create

`.github/workflows/ci.yml`: GitHub Actions workflow definition.

```yaml
name: CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

permissions:
  contents: read
  pull-requests: write
  checks: write
  id-token: write

jobs:
  unit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
          cache: pip
      - run: pip install -r requirements.txt
      - run: bash scripts/ci.sh

  integration:
    runs-on: ubuntu-latest
    needs: unit
    concurrency:
      group: integration-testbed-shared
      cancel-in-progress: false
    steps:
      - uses: actions/checkout@v4

      - id: testbed-token
        uses: actions/create-github-app-token@v1
        with:
          app-id: ${{ secrets.TESTBED_APP_ID }}
          private-key: ${{ secrets.TESTBED_APP_PRIVATE_KEY }}
          owner: AlexBomber12
          repositories: pipeline-orchestrator-testbed

      - name: Verify gh auth
        env:
          GH_TOKEN: ${{ steps.testbed-token.outputs.token }}
        run: gh auth status

      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
          cache: pip

      - run: pip install -r requirements.txt -r requirements-test.txt

      - run: playwright install --with-deps chromium

      - name: Build test stack image
        run: docker compose -f docker-compose.test.yml build

      - name: Initialize codex-auth-test volume
        run: bash scripts/init-test-volume.sh

      - name: Bring up test stack
        run: docker compose -f docker-compose.test.yml up -d --wait

      - name: Run e2e tests
        env:
          GH_TOKEN: ${{ steps.testbed-token.outputs.token }}
        run: pytest tests/e2e/ -v

      - name: Capture stack logs on failure
        if: failure()
        run: |
          docker compose -f docker-compose.test.yml logs > stack-logs.txt 2>&1 || true
          cat stack-logs.txt || true

      - name: Upload evidence artifact
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: e2e-evidence
          path: |
            tests/e2e/evidence/
            stack-logs.txt
          retention-days: 14
          if-no-files-found: ignore

      - name: Tear down test stack
        if: always()
        run: docker compose -f docker-compose.test.yml down -v
```

Important details for the implementer:
- The top-level concurrency group `${{ github.workflow }}-${{ github.ref }}` with `cancel-in-progress: true` means a new push to the same PR cancels the previous workflow run. This saves runner minutes when you push fixes.
- The integration job's nested concurrency block with group `integration-testbed-shared` and `cancel-in-progress: false` serializes integration runs across ALL PRs and pushes because they share one testbed repository. Second PR's integration job WAITS for first to complete; it does not cancel the first. This prevents two integration jobs from racing on the same testbed.
- `permissions: id-token: write` is required by actions/create-github-app-token for OIDC token exchange.
- `GH_TOKEN` env at the pytest step uses the App-generated testbed-scoped token, NOT the workflow's default GITHUB_TOKEN. The shim script invokes gh against the testbed repo and needs that token. Setting GH_TOKEN as env makes gh CLI auto-use it without an explicit `gh auth login` step.
- The `--with-deps` flag on `playwright install` installs system dependencies (libraries needed by Chromium). On ubuntu-latest runners this is cheap; on minimal images it would matter more.
- The `--wait` flag on `docker compose up -d` blocks until services pass their healthchecks. This requires Compose v2.20 or later, which ubuntu-latest runners have.
- The `if: always()` on tear-down ensures stack is cleaned up even if pytest failed, preventing leaked containers in the GHA cache.
- The artifact upload uses `if-no-files-found: ignore` because tests/e2e/evidence/ may be empty if no Playwright screenshots were taken.

## Success criteria

1. `.github/workflows/ci.yml` is syntactically valid YAML. Verifiable by `python -c "import yaml; yaml.safe_load(open('.github/workflows/ci.yml'))"` exiting 0.

2. The workflow defines exactly two jobs at the top level of the `jobs:` map: `unit` and `integration`.

3. The unit job has steps for checkout, setup-python, pip install, and bash scripts/ci.sh in that order.

4. The integration job has `needs: unit` so it runs only if unit passed, saving runner minutes on unit failures.

5. The integration job has a nested `concurrency:` block with `group: integration-testbed-shared` and `cancel-in-progress: false`. This is the global lock across all PRs.

6. The integration job uses `actions/create-github-app-token@v1` referencing `secrets.TESTBED_APP_ID` and `secrets.TESTBED_APP_PRIVATE_KEY`.

7. The integration job's pytest step has `GH_TOKEN` env set to the App token output (`steps.testbed-token.outputs.token`), NOT the default `secrets.GITHUB_TOKEN`.

8. The integration job's tear-down step uses `if: always()` so it runs even on test failure.

9. The integration job uploads an artifact named `e2e-evidence` containing tests/e2e/evidence/ and stack-logs.txt with retention 14 days.

10. The PR description explicitly states: "This PR's own integration job will fail on first run because the GitHub App is not yet configured. After merge, complete the GitHub App setup procedure documented in PR-155b's docs/ci-setup.md, then enable required checks via repo Settings → Branches. From the next PR onwards, both unit and integration are enforced as required checks."

11. scripts/ci.sh exits 0. ruff check src tests passes. No Python or other file changes outside the workflow YAML.
