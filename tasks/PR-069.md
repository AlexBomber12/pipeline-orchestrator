# PR-069: BUG-1 - no FIX when CI is PENDING

Branch: pr-069-no-fix-ci-pending
- Type: bugfix
- Complexity: low
- Depends on: none
- Priority: 1
- Coder: any

## Problem

In handle_watch, when review=CHANGES_REQUESTED and CI=PENDING, daemon enters FIX immediately. Codex Review check run may still be in progress. This wastes a FIX cycle and posts a duplicate @codex review.

Observed: Codex posts "no major issues" comment while its check run is still PENDING. Daemon sees CHANGES_REQUESTED (from the comment state), enters FIX, Claude makes unnecessary changes, daemon posts second @codex review.

## Scope

Add CI=PENDING guard before the feedback check in watch.py. If CI is still pending, wait regardless of review status.

## Changes

### src/daemon/handlers/watch.py

Before line 81 (`elif review == ReviewStatus.CHANGES_REQUESTED:`), add:

```python
elif ci == CIStatus.PENDING:
    pass
```

This makes the daemon fall through to the timeout/waiting log when CI has not resolved yet. FIX only triggers after CI settles to SUCCESS or FAILURE.

### tests/test_runner.py

Add:
- test_watch_no_fix_when_ci_pending_and_changes_requested: ci=PENDING, review=CHANGES_REQUESTED, expect state stays WATCH
- test_watch_fix_when_ci_success_and_changes_requested: ci=SUCCESS, review=CHANGES_REQUESTED, new feedback, expect FIX

## Success criteria
- scripts/ci.sh exits 0
- All tests pass
- Daemon does not enter FIX when CI=PENDING regardless of review status
- Daemon still enters FIX when CI=SUCCESS and review=CHANGES_REQUESTED with new feedback
