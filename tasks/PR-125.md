# PR-125: Auto-ready draft PRs before merge

Branch: pr-125-draft-pr-auto-ready
- Type: bugfix
- Complexity: low
- Depends on: none
- Priority: 2
- Coder: any

## Problem

Codex CLI occasionally creates pull requests as drafts. When the merge handler attempts `gh pr merge {number} --squash --delete-branch`, GitHub rejects the merge with `GraphQL: Pull Request is still a draft (mergePullRequest)`.

Without intervention the daemon goes to ERROR, then `diagnose_error` recommends FIX which cannot fix the draft status from code, then the cycle repeats. The operator has to manually run `gh pr ready {number}` to unblock.

The fix is simple: the daemon should call `gh pr ready {number}` immediately before `gh pr merge`. The `gh pr ready` command is idempotent - it exits 0 if the PR is already ready, and only has side effects if the PR is a draft. So it is always safe to call.

## Scope

Locate the place in the merge handler where `gh pr merge` is invoked. Add a `gh pr ready {pr_number}` call immediately before it. Swallow errors from `gh pr ready` silently - it is a best-effort prep step; if it fails for any reason other than the PR already being ready, the subsequent `gh pr merge` will surface the actual issue.

Do not change any other merge logic. Do not add new config flags.

## Files to modify

- `src/daemon/handlers/merge.py`: add `gh pr ready {pr_number}` call just before the existing `gh pr merge ...` invocation. Ignore non-zero exit codes from `gh pr ready` (log as debug, do not log as error).
- Tests in `tests/test_runner.py` (or wherever merge-handler tests live): add a test that verifies `gh pr ready` is called before `gh pr merge` in the happy path, and that a failure of `gh pr ready` does not block the merge attempt.

## Success criteria

- New test covers the happy-path ordering and the ignore-failure case.
- `pytest tests/test_runner.py -q` exits 0.
