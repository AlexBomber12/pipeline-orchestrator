## Problem

PR-153a established the Python directory structure under tests/e2e/. This PR adds the docker compose test stack: an isolated set of three services (web-test, daemon-test, redis-test) running on port 18800, completely separate from the production stack on 8800. Both stacks can run concurrently on the same host without conflict because they use different container names, different network names, different bind mount paths, and different host ports.

The test stack is not yet usable end-to-end at this PR's merge: there is no test config (PR-153c adds config.test.yml), no test dependencies declared (PR-153c), and no behavioral tests (PR-154 series). What this PR delivers is the docker-compose.test.yml file and the host-side helper script that initializes the named volume permission. After this PR merges, `docker compose -f docker-compose.test.yml up -d` should bring up three running containers but they will fail to do useful work because config.test.yml is absent. PR-153c completes the loop.

## Scope

Add docker-compose.test.yml at repo root. Add scripts/init-test-volume.sh helper that performs the codex-auth-test named volume chown trick (UID 1000) once. No source code changes. No config files. No dependency changes.

The compose file uses the existing Dockerfile. Both web-test and daemon-test reuse the same image (built once per docker compose build invocation). They differ only in command and port mapping.

## Files to create

`docker-compose.test.yml`: at repo root. YAML file with services, volumes, and networks blocks.

Top-level structure:

```
services:
  redis-test:
    image: redis:7-alpine
    container_name: pipeline-orchestrator-redis-test
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 3

  web-test:
    build: .
    container_name: pipeline-orchestrator-web-test
    command: bash scripts/entrypoint.sh python -m uvicorn src.web.app:app --host 0.0.0.0 --port 8000
    depends_on:
      redis-test:
        condition: service_healthy
    ports:
      - "${TEST_WEB_PORT:-18800}:8000"
    volumes:
      - ./tests/e2e/data:/data
      - codex-auth-test:/data/auth/.codex
    environment:
      REDIS_URL: redis://redis-test:6379/0
      CLAUDE_CONFIG_DIR: /data/auth/claude
      GH_CONFIG_DIR: /data/auth/gh
      HOME: /data/auth
      PO_CONFIG_PATH: /app/config.test.yml

  daemon-test:
    build: .
    container_name: pipeline-orchestrator-daemon-test
    command: bash scripts/entrypoint.sh python -m src.daemon.main
    depends_on:
      redis-test:
        condition: service_healthy
    volumes:
      - ./tests/e2e/data:/data
      - codex-auth-test:/data/auth/.codex
    environment:
      REDIS_URL: redis://redis-test:6379/0
      CLAUDE_CONFIG_DIR: /data/auth/claude
      GH_CONFIG_DIR: /data/auth/gh
      HOME: /data/auth
      PO_CONFIG_PATH: /app/config.test.yml

volumes:
  codex-auth-test:

networks:
  default:
    name: pipeline-orchestrator-test-net
```

Important notes for the implementer:
- Indentation matters for YAML; use 2-space indentation consistently.
- The TEST_WEB_PORT env var with default 18800 allows override on the rare case 18800 is already bound on the host.
- No mem_limit or cpu_limit overrides; inherit Docker defaults to keep configuration minimal.
- The bind mount for /app is intentionally absent on web-test and daemon-test in this stack. Production docker-compose.yml mounts ./:/app for fast reload during development; the test stack instead relies on the image's COPY layer (which has /app baked in at build time) to avoid file-watcher overhead. If a developer wants live reload during e2e debug, they can rebuild the image. PR-155c documents this tradeoff.
- The codex-auth-test named volume needs initialization to UID 1000 before the daemon can write into /data/auth/.codex. That initialization is automated in scripts/init-test-volume.sh.

`scripts/init-test-volume.sh`: bash script with `set -euo pipefail`. Single purpose: ensure the codex-auth-test named volume exists and is owned by UID 1000. Idempotent. Safe to run multiple times.

Body:
```
#!/usr/bin/env bash
set -euo pipefail

VOLUME_NAME="${VOLUME_NAME:-pipeline-orchestrator_codex-auth-test}"

if ! docker volume inspect "$VOLUME_NAME" >/dev/null 2>&1; then
  docker volume create "$VOLUME_NAME" >/dev/null
fi

docker run --rm -v "${VOLUME_NAME}:/target" alpine chown -R 1000:1000 /target

echo "OK: $VOLUME_NAME owned by 1000:1000"
```

The volume name prefix `pipeline-orchestrator_` matches Docker Compose's default project name derivation from the directory name (`pipeline-orchestrator`). If the project name differs (e.g. via COMPOSE_PROJECT_NAME env var), the script accepts override via VOLUME_NAME env var. Default covers the standard case.

Make the script executable: `chmod +x scripts/init-test-volume.sh`. The mode bit must be in the git index (use `git update-index --chmod=+x scripts/init-test-volume.sh` if necessary).

## Success criteria

1. `docker compose -f docker-compose.test.yml config` exits 0 and prints the resolved configuration with no errors. This validates YAML syntax and compose schema compliance.

2. `scripts/init-test-volume.sh` exits 0 on a fresh host (volume gets created). Re-running exits 0 (idempotent: volume exists, chown re-applied).

3. After running scripts/init-test-volume.sh, `docker volume inspect pipeline-orchestrator_codex-auth-test` shows the volume exists.

4. `docker compose -f docker-compose.test.yml build` exits 0 (image builds successfully). This may take 5 to 10 minutes on first invocation since the Dockerfile installs Node, npm packages, and Python packages.

5. With the prod stack running on port 8800, `docker compose -f docker-compose.test.yml up -d` succeeds and creates three containers named pipeline-orchestrator-web-test, pipeline-orchestrator-daemon-test, pipeline-orchestrator-redis-test. The two stacks coexist on the same host.

6. After up -d, `curl -sS http://localhost:18800/healthz` (or the equivalent healthcheck endpoint; if /healthz does not exist, use /api/states which returns 200 even without repos configured) returns HTTP 200, but with empty or error response body because config.test.yml does not yet exist (PR-153c adds it). This is acceptable for PR-153b.

7. `docker compose -f docker-compose.test.yml down -v` cleanly removes all three containers and the codex-auth-test volume. The prod stack on port 8800 is unaffected.

8. The two stacks use different docker network names: `pipeline-orchestrator-test-net` for the test stack vs. the default network name for the prod stack. `docker network ls | grep pipeline` shows both.

9. `git status` after running `up -d` and `down -v` shows no tracked-file changes. `tests/e2e/data/` may have content locally but is gitignored (gitignore from PR-153a).

10. scripts/ci.sh exits 0 unchanged. ruff check src tests passes (no Python changes in this PR). The shellcheck lint of init-test-volume.sh, if run manually, passes.
