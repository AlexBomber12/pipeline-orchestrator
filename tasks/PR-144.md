# PR-144: Bounded subprocess wait after TimeoutError and CancelledError

Branch: pr-144-cancel-wait-timeout
- Type: bugfix
- Complexity: medium
- Depends on: none
- Priority: 1
- Coder: any

## Problem

Both async coder CLI wrappers (`src/claude_cli.py` and `src/codex_cli.py`) have the same cleanup pattern in two except branches - `asyncio.TimeoutError` and `asyncio.CancelledError`. Both branches call `proc.kill()` followed by unbounded `await proc.wait()`. If the child process is in a zombie state or blocks past SIGKILL, `proc.wait()` hangs forever. The daemon task that was cancelled or timed out never returns, the state machine freezes in CODING or FIX, and only `docker compose restart daemon` recovers.

This becomes critical once PR-145 (FIX iteration cap) lands. Even though PR-145's cap check is at iteration boundaries (no mid-work cancel), Pause and Stop paths in the daemon still cancel coder tasks via `_monitor_stop_request` and `_terminate_current_coder`. A zombie wait in the CLI wrapper would block those cancellations from completing cleanly.

## Scope

Wrap both post-kill `proc.wait()` calls in `asyncio.wait_for(..., timeout=5)`. On timeout log a warning and proceed: re-raise original exception for the CancelledError path, return the timeout error tuple `(-1, "", "Timeout after {timeout}s")` for the TimeoutError path. The goal is bounded cleanup latency, not guaranteed reap. Container cgroup teardown on daemon shutdown will collect any truly unreapable process.

Both files have near-identical except blocks. Apply the same pattern consistently. Do not extract a shared helper in this PR - out of scope.

Expected diff size: roughly 6 to 10 lines changed per file per except block, 4 blocks total (2 per file).
